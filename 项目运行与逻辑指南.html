💡 这个项目是什么
铁矿石库存预测 + agent 自主研究报告 + 知识库 RAG 三合一：LightGBM 预测库存涨跌方向（MDA 84.91%），agent「研究团队」自动查预测数据和知识库里的过往报告，写成分析报告。
三块的关系：数据 + 文档 → agent → 报告
① 建模流水线（01~07） · 离线训练 LightGBM，产出预测结果到 07/outputs/。跑一次存盘
② agent 服务（agent_service + mcp_server） · LangGraph 研究团队，查资料写报告。:5000 + :17000
③ 知识库服务（knowledge_service） · RAG 检索过往文档，被 agent 调用。:8092

agent 是「中间人」：不存数据，只查现成的预测（左路）和知识库文档（右路），综合成报告。

① 客户端发请求

客户端（generate_report.py 或前端网页）向 :5000 发一个 POST，把问题和几个开关打包进去，然后用 SSE（Server-Sent Events）流式接收--不用等全部生成完，agent 边研究边把过程一行行推过来。

def generate_report(question, timeout=600):
    payload = {
        "messages": [{"role": "user", "content": question}],
        "agent_type": "default",
        "auto_accepted_plan": True,          # 自动接受计划，不中断等人审
        "enable_background_investigation": False,
        "enable_clarification": False,       # 不多轮澄清
        "locale": "zh-CN", "max_step_num": 4,
    }
    with httpx.Client(timeout=timeout) as client:
        with client.stream("POST", AGENT_URL, json=payload) as resp:
            for line in resp.iter_lines():        # SSE 一行行读
                if line.startswith("event: "):  event_type = line[7:].strip()
                elif line.startswith("data: "):
                    if line[6:] == "[DONE]": break   # 流结束标志
                    handle_event(event_type, json.loads(line[6:]), ...)

为什么用 bind_tools 而不是写死跳转？

coordinator 把「去哪」也交给 LLM 决定。LLM 读问题后，如果是闲聊可能直接回（去 END），如果要研究就调 handoff_to_planner。这是 LangGraph 的 工具即路由模式：LLM 调哪个 handoff 工具，图就跳到对应节点。

消息过滤的细节

coordinator 会过滤掉 name=='planner' 或 'deepresearch' 的 AI 消息（不让自己看到规划思路），也过滤自己带 tool_calls 的消息（路由指令不暴露给下一轮）。只保留干净的对话上下文。这是为了避免上下文污染。

反馈式轮询（核心机制）

不是一次把所有 step 派完，而是：派一个/一批 → 回调度室 → 重算就绪 → 派下一个，循环到全部 execution_res 都填满。这样后面的 step 能看到前面 step 的发现，依赖关系靠这轮询兑现。这也是为什么 researcher 做完要 goto="research_team" 回来。

为什么节点是空的？

LangGraph 的节点不一定要干活，可以只是「编排点」。research_team 就是这样一个点--它存在的意义是让多条边汇聚到这里、由条件边决定分流。真正的状态判断在 continue_to_running_research_team 和 _execute_agent_step 里。

关键：直接读文件，不算预测

agent 不训练模型、不算预测。它只是读 07 目录里已经算好的结果。所以建模流水线不用每次跑，除非你重训了模型想用新结果。这是「离线产数据 + 在线查数据」的解耦。

返回的是文本，不是结构化数据

fetch_data 把多个文件内容拼成一段带标签的文本（【标的】…【模型评估】…）返回。agent 读这段文本理解数据，再写进报告。MCP 工具的返回都是字符串。

研究员调 fetch_knowledge。这个工具不读文件，而是用 httpx 转发到知识库服务 :8092，让 knowledge_service 跑 RAG 检索，返回最相关的过往文档片段。这是知识库真正被用到的时刻。

fetch_knowledge 的实现

@mcp.tool()
def fetch_knowledge(query, topK=5, reRank=1) -> str:
    """知识库检索：过往分析报告/笔记/项目文档。"""
    r = httpx.post(
        "http://127.0.0.1:8092/knowledge-base/report/search",
        json={"content": query, "topK": topK, "reRank": reRank, "mode": "mix"},
        timeout=120,
    )
    data = r.json().get("data", [])
    # 拼成「【知识库检索 N 条】...」文本返回给 agent
    for i, h in enumerate(data, 1):
        parts.append(f"{i}. [{h['fileName']}] {h['content'][:...]}...")
    return "\n\n".join(parts)

解耦：agent 不直接调知识库

agent 调的是 mcp_server 的 fetch_knowledge，由 mcp_server 转发到 :8092。所以从 agent 视角，知识库只是「fetch_knowledge 返回的文档片段」；知识库可以独立优化（换 embedding、调 chunk_size）而不动 agent。

    for observation in state.get("observations", []):
        invoke_messages.append(HumanMessage(content=f"观察发现:\n{observation}"))

    response = await get_llm_by_type(AGENT_LLM_MAP["reporter"]).ainvoke(invoke_messages)
    # ChatAnthropic 可能返回 content 列表（thinking+text），提取 text
    if isinstance(response.content, list):
        response.content = "".join(b["text"] for b in response.content if b.get("type")=="text")
    return {"final_report": response.content, "messages": response}

报告不是一下子出来的，是流出来的

reporter 用 ainvoke（非流式）生成完整报告，但整条链路的 SSE 是流式的--reporter 的输出会作为 message_chunk 事件被切成 token 推给客户端，客户端边收边拼边打印。所以你看到报告是「一段段冒出来」的。

客户端收尾

客户端收到 [DONE] 后，把拼好的 report_chunks 存成 outputs/report_<时间戳>.md，打印「✓ 报告已保存」+ 工具调用次数，然后退出。知识库指南里有完整报告样例可看。

📖 术语表

MCP（Model Context Protocol）
把「外部能力」标准化暴露给 LLM 的协议。mcp_server 用 MCP 把 8 个工具暴露给 agent，agent 不关心数据在文件还是数据库，只管调工具。
LangGraph
把 LLM 工作流组织成「图」的框架，节点是步骤、边是流转。支持条件路由、并行、循环。本项目的 coordinator→planner→research_team→researcher→reporter 就是张图。
ReAct
Reason+Act，LLM 边推理边调工具的循环：思考→调工具→看结果→再思考。researcher 就是 ReAct agent，决定调哪个工具、调几次。
SSE（Server-Sent Events）
服务端推流。agent 用 SSE 把研究过程和报告流式推给客户端，不用等全部生成完才返回，能边跑边看。事件格式 event: 类型\ndata: {json}，[DONE] 结束。
RAG（检索增强生成）
先从知识库检索相关片段，再喂给 LLM 生成。knowledge_service 做「检索」那半，生成那半在 agent。
bi-encoder / cross-encoder
bi-encoder：query 和 doc 各自编码成向量算距离，快但浅（bge-m3，用于召回）；cross-encoder：query+doc 拼一起打分，慢但准（bge-reranker，用于重排）。
RRF（Reciprocal Rank Fusion）
按排名融合多路检索结果，分数=Σ 1/(k+rank)，只看排名不看分数量纲。把 KNN 和 BM25 两份排名合成一份。
Chroma
轻量向量数据库，本地文件、零运维。存 237 个 chunk 的 1024 维向量，cosine 相似度做 KNN。
BM25
经典关键词检索，基于 TF+IDF。中文要先 jieba 分词。和向量检索互补。
MDA（方向准确率）
预测方向（涨/跌）与实际一致的比例。主标的测试集 MDA=84.91%，是模型核心指标。
Optuna
超参自动优化框架，贝叶斯优化（TPE）。本项目跑 200 次试验找最优 LightGBM 超参。
GLM / 火山方舟
本项目 LLM 用智谱 GLM（glm-5-2-260617），通过火山方舟的 Anthropic 兼容接口调用。key 在 .env.dev。
