Q1 · 提问后，系统怎么开始干活的？

驱动问题
我敲完一句话回车，是谁第一个接住？它怎么决定接下来干嘛？
💡 一句话答案
请求进到 agent_service，LangGraph 图的入口节点 coordinator（前台）接住，它给 LLM 几个"转交工具"，LLM 选了 handoff_to_planner，于是流程跳到 planner。

什么是 LangGraph「图」

LangGraph 把一个任务画成一张有向图：每个节点（node）是一个干活的函数，边（edge）决定执行顺序。图有一个入口 START，一个出口 END。看 builder.py 怎么搭这张图：

# agent_service/src/graph/deepresearch/builder.py
builder = StateGraph(State)
builder.add_edge(START, "coordinator")          # 一进门就找前台
builder.add_node("coordinator", coordinator_node)
builder.add_node("planner", planner_node)
builder.add_node("researcher", researcher_node)
builder.add_node("parallel_researcher", parallel_researcher_node)
builder.add_node("reporter", reporter_node)
...
builder.add_edge("parallel_researcher", "research_team")  # 调研完回调度
builder.add_edge("reporter", END)              # 写完报告结束

注意：边不全是写死的。research_team 后面跟的是条件边——由一个函数根据当前进度决定下一步去哪（Q4 会细讲）。

前台 coordinator 怎么"决定转给谁"

coordinator 不是用 if/else 硬编码，而是给 LLM 一组"转交工具"，让 LLM 自己选。看 nodes.py：

# coordinator_node 里
tools = [handoff_to_planner, handoff_after_clarification, ...]
response = await get_llm_by_type(...).bind_tools(tools).ainvoke(messages)

if response.tool_calls:          # LLM 选了某个转交工具
    for tool_call in response.tool_calls:
        if tool_call["name"] == "handoff_to_planner":
            goto = "planner"     # → 跳到项目经理

而这个 handoff_to_planner 工具其实啥都不干，它只是个"信号"：

# nodes.py
@tool
def handoff_to_planner(research_topic: str, locale: str):
    """Handoff to planner agent to do plan."""
    return   # 不返回任何东西，纯粹是让 LLM 表达"我要转给 planner"
为什么用"空工具"而不是 if/else？

因为 LLM 擅长理解语义。你说"帮我分析库存"，LLM 读得懂该转给研究流程；你说"你好"，LLM 可以选择不转交、直接回你。把决策权交给 LLM，比写死规则灵活得多。这是 agent 区别于普通程序的关键：流程不是人写死的，是模型动态选的。

动手实验 1 · 观察 coordinator 的转交
  1. 确保终端 1、2 还开着。
  2. 终端 3 跑一次报告生成，盯着终端 2（agent_service）的日志。
  3. 你会看到类似 [工具调用] agent=coordinator tools=['handoff_to_planner']——这就是前台选了"转给项目经理"。
  4. 改个提问试试：编辑 generate_report.py 末尾 default_q，换成 "你好，你能做什么？"，再跑。观察 coordinator 这次是否还转给 planner（很可能不转，直接回你）。
你会观察到：不同提问，coordinator 的 tool_calls 不一样。流程是模型选的，不是写死的。
3 题自测（点开看答案）
LangGraph 图里，"节点"和"边"分别是什么？
看答案
节点是一个干活的函数（如 coordinator_node）；边决定执行顺序（如 START→coordinator→planner）。条件边则由函数运行时决定下一站。
coordinator 为什么用"工具"来决定转交，而不是 if/else？
看答案
因为转交决策依赖语义理解（用户到底想干嘛），交给 LLM 选工具比硬编码规则更灵活。这是 agent 的核心特征：流程由模型动态决定。
handoff_to_planner 工具内部有逻辑吗？为什么这样设计？
看答案
没有，它直接 return。它只是个信号——让 LLM 用"调用这个工具"来表达"我要转给 planner"，代码侧检测到这个 tool_call 就跳转。把意图和执行解耦。

Q2 · 系统怎么知道"研究什么、分几步"？

驱动问题
前台转给项目经理后，计划是哪来的？谁决定要查哪几样东西？
💡 一句话答案
planner 让 LLM 流式输出一段 JSON 计划（几个步骤、每步要查啥），代码解析成 Plan 对象。因为开了 auto_accepted_plan，计划自动通过、不等人工审。

planner 流式拼出一段 JSON

planner 让 LLM 边想边吐字，把每个 chunk 拼成完整字符串，再当 JSON 解析：

# nodes.py · planner_node
full_response = ""
async for chunk in llm.astream(compressed_state["messages"]):
    full_response += _content_to_text(chunk.content)   # 拼成完整 JSON 文本

curr_plan = json.loads(repair_json_output(full_response))  # 解析成 dict
curr_plan = Plan.model_validate(curr_plan)                 # 校验成结构化计划

解析出来的 Plan 大致长这样（真实跑出来的 4 步计划）：

{
  "steps": [
    {"title": "项目背景与建模流程",  "step_type": "RESEARCH"},
    {"title": "预测结果与模型评估指标", "step_type": "RESEARCH"},
    {"title": "模型选择与超参优化结果", "step_type": "RESEARCH"},
    {"title": "测试集预测结果分析",   "step_type": "RESEARCH"}
  ]
}

这 4 步就是接下来 researcher 要一个个查的题目。计划的步数由请求里的 max_step_num 限制。

计划要不要人审？

LangGraph 原本支持"计划生成后暂停，等用户确认再继续"（human-in-the-loop）。但我们客户端请求里设了：

# generate_report.py 的 payload
payload = {
    "auto_accepted_plan": True,        # 自动通过，不等人工审
    "enable_clarification": False,    # 不多轮追问澄清
    "max_step_num": 4,
    ...
}

所以 human_feedback 节点直接放行，流程不停顿——适合自动出报告。如果你要做交互式产品，可以关掉这两个，让用户审计划。

动手实验 2 · 改计划步数
  1. 打开 generate_report.py，把 "max_step_num": 4 改成 2。
  2. 终端 3 重跑（终端 1、2 不用动）。
  3. 盯终端 2 日志，找 [planner_node] max_step_num before prompt render: 2 这行。
  4. 再看后面 Executing step: ... 的行数——从 4 步变成 2 步。
你会观察到：max_step_num 直接控制 planner 拆几步。步数少 → 研究粗 → 报告浅；步数多 → 更细但更慢、更费 token。
3 题自测
planner 输出的是一段文本，代码怎么把它变成"能执行的步骤"？
看答案
先用 json.loads 把 JSON 文本解析成 dict，再用 Plan.model_validate 校验并转成结构化的 Plan 对象（pydantic）。校验失败会走降级逻辑。
为什么要 async for chunk in llm.astream(...) 而不是一次性 ainvoke？
看答案
流式 astream 边生成边拿，可以更早开始处理、避免长文本一次性返回的等待。这里主要拼完整字符串；在 reporter 那里流式还能让用户实时看到字（Q5）。
auto_accepted_plan=True 关掉会怎样？
看答案
计划生成后流程会在 human_feedback 节点暂停（interrupt），等用户确认计划才继续。适合交互式产品，不适合自动出报告。

Q3 · researcher 怎么"知道"去查铁矿石数据？又怎么查到的？

驱动问题
researcher 是个通用 AI，它怎么知道这项目有"库存预测结果"可查？数据又是从哪冒出来的？
💡 一句话答案
靠 MCP：我们自建了一个 MCP server（档案室），把复现2 的数据包成几个"工具"；配置文件告诉 agent "researcher 能用这些工具"；researcher 就像调函数一样调它们拿数据。

什么是 MCP

MCP（Model Context Protocol）是一个让 AI 调用外部工具/数据源的协议。你可以理解为：给 AI 装一个"USB 接口"，插上不同的工具它就能用。这里我们把"读复现2 建模产出"做成了一个 MCP server，AI 通过它查数据。

为什么不直接在代码里写个函数让 AI 调？因为 MCP 是独立进程、统一协议：工具和 agent 解耦，换数据源不用改 agent，多个 agent 能共用同一份工具。（Q7 会对比"本地 tool vs MCP"。）

自建 MCP server 长啥样

用 mcp 库的 FastMCP，写法和 Flask 很像——每个 @mcp.tool() 就是一个 AI 可调的工具：

# mcp_server/server.py
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("iron-ore-reproduction-mcp", host="127.0.0.1", port=17000)

@mcp.tool()
def fetch_data(query: str) -> str:
    """查询铁矿石库存预测的核心数据：标的、评估指标、测试集预测、特征体系。"""
    parts = []
    parts.append("【标的】ID00186052 铁矿石：进口：库存：45个港口（周度）")
    parts.append(_read_text(OUT_07 / "final_evaluation_weekly.json"))  # 读真实评估
    parts.append(_read_csv_tail(OUT_07 / "test_predictions_weekly.csv")) # 读预测
    return "\n\n".join(parts)

if __name__ == "__main__":
    mcp.run(transport="streamable-http")   # 跑成 HTTP 服务

工具的docstring 就是给 AI 看的说明书——AI 靠它判断"这个工具该不该调"。所以 docstring 要写清楚"能查什么、什么时候调"。

agent 怎么知道 researcher 能用哪些工具

在 .env.dev 的 MCP_DEFAULT_SETTINGS 里配了一张"工具表"，写明每个角色能用哪些工具、MCP server 在哪：

# .env.dev（简化展示）
MCP_DEFAULT_SETTINGS='{"servers":{"xmschain-mcp-tools":{
  "enabled_tools":[
    {"node":"researcher","tools":["fetch_data","fetch_news","model_choice","model_predict","draw_chart"]},
    {"node":"background","tools":["fetch_background"]}
  ],
  "transport":"streamable_http",
  "url":"http://127.0.0.1:17000/mcp"   # 指向自建 MCP server
}}}'

agent 启动 researcher 时，merge_mcp_tools 会按这张表，连到 17000 端口把工具加载进来，注入给 ReAct agent：

# nodes.py · _setup_and_execute_agent_step
enable_mcp_servers, loaded_tools = await merge_mcp_tools(
    agent_name, configurable.mcp_settings, default_tools)
agent = create_agent(graph_name, agent_name, loaded_tools, prompt_template)
关键链条（记住这条线）

mcp_server 暴露工具 → .env.dev 配工具表和地址 → merge_mcp_tools 加载注入 → researcher 的 ReAct agent 拿到工具 → LLM 决定调哪个 → 工具读复现2 文件返回数据。任何一环断掉，researcher 就查不到数据。

动手实验 3 · 关掉档案室看报错
  1. 终端 3 先正常跑一次，确认日志里能看到 [工具调用] agent=parallel_researcher tools=['fetch_data'] 和 └ 工具返回: 【标的】ID00186052...——说明数据查到了。
  2. 去终端 1 按 Ctrl+C 关掉 MCP server（档案室关门）。
  3. 终端 3 再跑一次报告生成。
  4. 盯终端 2 日志：researcher 调工具时会失败/超时，报告会缺少真实数据（或流程报错）。
  5. 实验完，终端 1 重新跑命令把 MCP server 开回来。
你会观察到：MCP server 是数据的唯一来源。关掉它，researcher 再聪明也拿不到铁矿石数据。这帮你理解 MCP 的"桥梁"角色。
3 题自测
researcher 怎么知道有 fetch_data 这个工具可以用？
看答案
两步：① MCP server 用 @mcp.tool() 暴露工具，docstring 描述用途；② .env.dev 的工具表声明 researcher 能用哪些，merge_mcp_tools 加载注入。LLM 根据 docstring 判断何时调。
工具的 docstring 是写给谁看的？为什么要认真写？
看答案
写给 LLM 看。LLM 靠 docstring 判断"这个工具能干什么、我该不该调"。docstring 写得糊，LLM 就会乱调或不调。
MCP server 和 agent_service 是同一个程序吗？为什么要分开？
看答案
不是。MCP server 是独立进程（端口 17000），agent_service 是另一个（端口 5000）。分开是为了解耦：数据源可独立维护/复用，多个 agent 都能连同一个 MCP server，换数据源不用改 agent 代码。

Q4 · 查到的数据怎么变成"结论"传给写手？

驱动问题
researcher 查完一份数据，这结果存哪？写手怎么拿到所有步骤的结论？多步怎么并行？
💡 一句话答案
所有节点共享一块叫 state 的"白板"。researcher 把每步结论写进 execution_res；research_team 节点像个调度员，用条件边看白板进度，有多个就绪步骤就派 parallel_researcher 并行，全做完就交给 reporter。

state：团队共享白板

LangGraph 图里所有节点共享一个 state 对象。你可以把它想象成团队办公室墙上的一块白板：每个节点干完活，把结果写到白板上；下一个节点从白板上读。planner 写 current_plan，researcher 写 execution_res，reporter 读所有这些。

这是 LangGraph 的核心设计

节点之间不直接传参，而是通过 state 读写。这样节点解耦、可并行、可恢复（白板能存档）。state 定义在 types.py，是个带字段的类。

research_team：调度员 + 条件边

计划有 4 步，谁来决定"现在该做哪步、做完去哪"？是 research_team 节点配的那个条件函数：

# builder.py
builder.add_conditional_edges(
    "research_team",                       # 从 research_team 出发
    continue_to_running_research_team,       # 由这个函数决定下一站
    ["planner", "researcher", "parallel_researcher", "analyst", "reporter"],
)
builder.add_edge("parallel_researcher", "research_team")  # 跑完回调度
# nodes.py · continue_to_running_research_team（简化）
# 1. 全部步骤都做完了？
if all(step.execution_res is not None for step in plan.steps):
    return "planner"     # → 交给 planner 收尾/或 reporter

# 2. 找出"依赖已满足、还没做"的步骤
ready_research_steps = [...]
if len(ready_research_steps) > 1:
    return "parallel_researcher"   # 多个就绪 → 并行
if ready_research_steps:
    return "researcher"           # 单个 → 串行

这就是为什么日志里你会看到 4 个步骤是同时开始执行的——条件函数发现 4 步互不依赖，全部就绪，于是派 parallel_researcher 并行跑。

research_team调度
→
parallel_researcher4步并行
→
research_team再看白板
→
reporter全做完→写报告
动手实验 4 · 看并行与结论落地
  1. 终端 3 跑一次报告生成，盯终端 2 日志。
  2. 找 本次将进行递归的任务执行，共 4 个并行任务: ['项目背景与建模流程', '预测结果与模型评估指标', ...] 这行——确认 4 步并行。
  3. 找 4 行 Step '...' execution completed by researcher——每步完成后，结论写进了 state 的 execution_res。
  4. 最后找 Reporter write final report——写手这时才从白板读所有结论开始写。
你会观察到：researcher 不直接调 reporter，而是把结论写白板、回调度、调度看"全做完了"才放行给 reporter。这就是 state + 条件边的编排方式。
3 题自测
researcher 和 reporter 之间怎么传数据？
看答案
不直接传。researcher 把每步结论写进共享 state 的 execution_res；reporter 从 state 读所有结论。节点间通过 state 解耦。
4 个研究步骤为什么能并行？谁决定的？
看答案
continue_to_running_research_team 条件函数发现 4 步互不依赖、全部就绪，于是返回 parallel_researcher 触发并行。是运行时根据 state 动态决定的，不是写死。
条件边和普通边的区别？
看答案
普通边是写死的固定跳转（A→B）。条件边由一个函数在运行时返回下一站，能根据 state 灵活路由（A→B 或 A→C）。研究流程的"做完没/能否并行"判断全靠它。

Q5 · 报告怎么"边想边写"实时流到我屏幕？

驱动问题
写手写报告时，我不是等它写完才看到，而是一字一字往外冒。这个"流"是怎么实现的？
💡 一句话答案
靠 SSE（Server-Sent Events）：服务端把生成过程拆成一个个小事件（吐字/调工具/工具返回），通过 HTTP 长连接一条条推给客户端；客户端按 event: / data: 格式解析。

什么是 SSE

SSE 就是服务器主动往客户端"推"消息的协议。基于 HTTP 长连接，服务器不停往里写文本，浏览器/客户端边收边处理。这里用它把"AI 生成报告的过程"实时推出来，不用等整篇写完。

SSE 的格式非常简单，每条事件两行：

event: message_chunk
data: {"agent": "reporter", "content": "# 铁矿石库存预测..."}

event: tool_calls
data: {"agent": "researcher", "tool_calls": [{"name": "fetch_data"}]}

一个空行分隔每条事件。客户端逐行读：event: 行说"这是什么事件"，data: 行是 JSON 内容。

4 种事件类型

事件含义客户端怎么用
message_chunk模型吐了一小段文字如果 agent 是 reporter，拼进报告；否则是别的节点在"想"
tool_calls模型决定调某个工具打印 [工具调用] tools=[...]，让你看到 AI 在查啥
tool_call_result工具返回结果了打印前 80 字，看到查到的数据片段
error出错了打印错误信息

客户端怎么解析

# generate_report.py
for line in resp.iter_lines():
    if line.startswith("event: "):
        event_type = line[7:].strip()       # 取 "message_chunk"
    elif line.startswith("data: "):
        data_str = line[6:]
        if data_str == "[DONE]": break # 结束标记
        data = json.loads(data_str)          # 解析 JSON
        handle_event(event_type, data, ...)

# handle_event 里：只收 reporter 的字作为最终报告
if event_type == "message_chunk":
    if "reporter" in agent.lower():
        report_chunks.append(content)        # 拼报告
为什么只收 reporter 的字？

因为 planner、researcher 也会吐字（message_chunk），但那些是"思考过程"不是报告。只有写手 reporter 吐的才是要给用户的成品。靠事件里的 agent 字段区分。

动手实验 5 · 打印每条事件
  1. 打开 generate_report.py，在 handle_event 函数开头加一行：print(f"[事件] type={event_type} agent={data.get('agent','')}", flush=True)。
  2. 终端 3 重跑。
  3. 你会看到大量 [事件] type=message_chunk agent=... 和 type=tool_calls——实时看到 AI 一边想一边调工具一边写字。
  4. 注意 agent=reporter 的 message_chunk，就是报告正文在逐字流出。
你会观察到：报告不是一个整块返回的，是几百个 message_chunk 拼起来的。SSE 让你能"看着 AI 写"。
3 题自测
SSE 和普通 HTTP 请求的最大区别？
看答案
普通 HTTP 是"请求→一个完整响应→结束"。SSE 是长连接，服务器持续推多条事件，客户端边收边处理。适合流式生成场景。
客户端怎么从一堆 message_chunk 里挑出"报告正文"？
看答案
看每条事件的 agent 字段。只有 agent 包含 "reporter" 的 message_chunk 才是报告正文，其他节点的是思考过程。拼接到 report_chunks。
data: [DONE] 是什么意思？
看答案
流结束的标记。客户端收到它就 break 退出循环，结束接收。这是约定好的哨兵值。

Q6 · 联调崩了，怎么像侦探一样定位修复？

驱动问题
第一次联调直接报错。真实项目里 bug 怎么找、怎么修？
💡 一句话答案
联调踩了两个真实 bug：① planner 拼字符串时崩（LLM 返回的不是字符串是列表）；② 报告写到一半断了（token 不够）。靠读报错栈定位节点→理解根因→最小修复。

Bug 1 · TypeError: can only concatenate str (not "list") to str

报错现场
TypeError: can only concatenate str (not "list") to str
  File ".../nodes.py", line 431, in planner_node
    full_response += chunk.content
        ↑              ↑
       str            list: [{'thinking': 'The', 'type': 'thinking', 'index': 0}]

侦探步骤：报错栈直接指到 planner_node 第 431 行，full_response += chunk.content。full_response 是字符串，但 chunk.content 居然是个列表。为什么？

根因

我们用的 LLM 是火山方舟 GLM，走 Anthropic 协议（ChatAnthropic）。这个端点默认会返回 thinking 块（模型的思考过程），导致 content 是一个列表：[{'thinking':..., 'type':'thinking'}, {'text':..., 'type':'text'}]，而不是纯字符串。代码原本假设 content 是字符串，于是 str += list 崩了。

修复前

full_response += chunk.content
# chunk.content 是 list → 崩

修复后

full_response += _content_to_text(chunk.content)
# 提取 text 块、跳过 thinking 块
# nodes.py · 新增 helper
def _content_to_text(content):
    if isinstance(content, str): return content      # OpenAI 协议：本来就是字符串
    if isinstance(content, list):                 # Anthropic 协议：thinking+text 列表
        parts = []
        for block in content:
            if isinstance(block, dict) and block.get("type") == "text":
                parts.append(block.get("text", ""))   # 只取 text，跳过 thinking
        return "".join(parts)
    return str(content)
学到什么

① 报错栈是最好的向导，直接指到出问题的行。② 不同 LLM 协议返回结构不同，写兼容代码要考虑。③ thinking 块对人是噪音，提取时跳过。④ reporter、SSE 工具里同样的坑也要一并修（任何 += content 都得套 helper）。

Bug 2 · 报告写到一半突然断了

现象

报告在「七、结论」的表格中间戛然而止：| **绝对精度** | 中等 后面就没了，明显没写完。

侦探步骤：没报错，但输出被截断 → 典型的 max_tokens 不够。为什么不够？因为 thinking 块也占 token 配额——模型先"想"了一大段（消耗 token），留给"写正文"的额度就不够了。

修复前

.env.dev：LLM_MAX_TOKENS=4096

thinking 吃掉一大半，报告写不完。

修复后

.env.dev：LLM_MAX_TOKENS=16000

thinking + 完整报告都装得下。

学到什么

① "没报错但结果不完整"常是 token 截断。② 带 thinking 的模型，max_tokens 要算上思考的消耗。③ 改完 .env.dev 要重启 agent_service才生效（uvicorn 没开热重载）。

动手实验 6 · 复现 Bug 1（安全地）
  1. 打开 nodes.py，把 full_response += _content_to_text(chunk.content) 临时改回 full_response += chunk.content。
  2. 终端 2 按 Ctrl+C 关掉 agent_service，重新跑那条命令重启（改了代码必须重启）。
  3. 终端 3 跑报告生成。
  4. 终端 2 会出现那个 TypeError: can only concatenate str (not "list") to str 报错栈，指到 planner_node。
  5. 改回 _content_to_text，重启 agent_service，确认恢复正常。
你会观察到：一行代码差异，整个流程在 planner 就崩了。这就是为什么理解"content 是列表"这个根因这么重要。
3 题自测
为什么 ChatAnthropic 返回的 content 是列表而不是字符串？
看答案
因为火山方舟 GLM 的端点默认返回 thinking 块（思考过程），所以 content 是 [thinking 块, text 块] 的列表。OpenAI 协议默认不返回 thinking，content 是纯字符串。
报告没报错却写一半就停，最可能的原因？
看答案
max_tokens 截断。带 thinking 的模型，思考过程也占 token 配额，留给正文的额度不够。调大 LLM_MAX_TOKENS 并重启服务。
改了 src/ 代码或 .env.dev，为什么必须重启 agent_service？
看答案
因为 uvicorn 启动时没加 --reload，不会自动热重载。配置和代码在进程启动时加载一次，改了不重启不生效。

Q7 · 为什么搞这么复杂？不用 LangGraph/MCP 会怎样？

驱动问题
不就生成个报告吗，写个函数链调用 LLM 不就行了？为什么要 LangGraph + MCP 这么重？
💡 一句话答案
简单函数链能跑通最简场景，但没法并行、没法中断恢复、没法流式、工具和逻辑耦合。LangGraph 解决"流程编排"，MCP 解决"工具解耦"。复杂度换来的是工程能力。

简单函数链 vs LangGraph

简单函数链

def run(question):
    plan = llm_plan(question)
    for step in plan:
        data = llm_research(step)
    report = llm_report(...)
    return report
  • 顺序写死，不能并行4 步研究
  • 中途崩了没法恢复，得从头跑
  • 用户看不到过程，只能等最后结果
  • 流程一变就得改代码

LangGraph

  • 条件边 + parallel_researcher → 自动并行就绪步骤
  • checkpointer 存 state → 崩了能从断点恢复（本项目用 MemorySaver 简化了）
  • stream_mode → 全程流式吐事件
  • 节点/边配置化，流程易调整

本地 tool vs MCP

本地 tool（直接写函数注入）

  • 工具和 agent 同一个进程，耦合
  • 换数据源要改 agent 代码
  • 多个 agent 难共用同一套工具
  • 但：简单、快、无网络开销

MCP（独立 server + 协议）

  • 工具独立进程，解耦
  • 换数据源不动 agent
  • 多个 agent 共用一个 MCP server
  • 代价：多一个进程、网络调用
诚实提示

对你这个数据建模项目，完整 agent 服务（常驻 FastAPI + 多节点编排）形态上偏重。它的核心价值不是"必须这么做"，而是让你体验完整的 planner→researcher→reporter 自主研究闭环，理解 agent 架构怎么工作。真要做个轻量预测报告，一个脚本调 LLM 也行——但你学不到这些编排思想。

3 题自测
用简单函数链能不能实现这个项目？差什么？
看答案
能跑通最简版。但缺：并行研究（慢）、断点恢复（崩了重来）、流式输出（用户干等）、流程灵活调整。LangGraph 用复杂度换了这些工程能力。
什么场景下 MCP 的"解耦"特别值？
看答案
多个 agent 要共用同一套工具、或数据源由不同团队维护、或想不改 agent 代码换数据源时。如果就一个 agent + 固定数据，本地 tool 更简单。
checkpointer 是干嘛的？本项目为什么关掉了？
看答案
checkpointer 把 state 持久化（如存 Postgres），让流程能中断/恢复。本项目为免 Postgres 依赖，设 LANGGRAPH_CHECKPOINT_SAVER=False，用内存 MemorySaver，代价是重启就丢状态。够用于自动出报告。

Q8 · 自己搭 agent，和直接用 Claude Code 扫文件夹有啥区别？

驱动问题
有了 Claude Code 这种能扫整个文件夹、自己找文件的 AI，我干嘛还要费劲搭这套 agent？
💡 一句话答案
这是两个层次的选择。Claude Code 是放权给 AI（AI 自己设计流程、自己找文件），自建 agent 是约束 AI（人设计好流程，AI 只在决策点填智能）。前者灵活万能，后者稳定、可产品化、能给非技术人员用。

本质：谁在设计流程

直接用 Claude Code

流程也交给 AI。AI 自己决定读哪个文件、grep 什么、跑什么命令。人只下指令。

AI 既是工位，又是流程设计师。

灵活万能 不可预测

自建 agent

流程攥在人手里。人画好图（coordinator→planner→researcher→reporter），AI 只在「选工具/拆步/写报告」填智能。

AI 只是流水线上的智能工位。

稳定可控 只擅长预见到的事

分维度对比

维度你的 agent 项目Claude Code 扫文件夹
工具窄而专：fetch_data 直接吐结构化结果，AI 不知文件路径宽而通用：Read/Grep/Glob/Bash 扫任意位置
数据接入MCP 预先桥接，AI 调工具即得AI 自己 glob/grep 摸索项目布局
流程LangGraph 图固定编排，有条件边、并行无固定图，模型即流程，即兴生成
并行parallel_researcher 多步同时跑基本串行探索
可复现性同问题走同样节点，输出稳定每次路径可能不同，不可预期
部署形态常驻 FastAPI 服务，HTTP 可调，可嵌网页/多用户本地 CLI/IDE，单人交互，非服务
受众业务人员：只会问「生成报告」开发者：要会看代码判断结果
领域知识内化在 prompts 和工具 docstring通用知识，靠上下文临时注入
核心洞察：agent 是「约束工程」

搭 agent 不是「让 AI 更强」，而是把 AI 约束进可控流水线。约束体现在四层：① 流程约束（固定图）② 工具约束（窄 MCP）③ 知识约束（prompt 固化）④ 输出约束（结构化校验）。每一层都是用灵活性换可控性——工程就是把不确定性约束掉，让结果可预期。

约束换来什么 / 代价什么

约束换来

  • 稳定、可复现
  • 可并行、可恢复
  • 可产品化（给别人用）
  • 输出标准化
  • 领域知识固化
  • 可控不跑偏

约束的代价

  • 灵活性（只擅长预见到的任务）
  • 搭建成本（画图/写工具/调 prompt）
  • 维护成本（流程一变要改代码）
  • 不能跨领域即兴发挥

什么时候用哪个

场景选什么
个人偶尔看一次报告 / 探索陌生代码 / 调 bug / 跨领域未知问题直接用 Claude Code 零搭建，灵活
给别人用 / 重复定时跑 / 嵌入产品网页 / 多用户 / 输出要稳定标准自建 agent 约束化，可产品化
最大的意义：把「会用 AI」的门槛，从用户侧搬到开发者侧

直接用 Claude Code 出报告，用户要会装环境、看代码、写 prompt——门槛把 AI 问答关在开发者圈子里。自建 agent 把这些复杂度全吞进服务内部，业务人员只要一个输入框问一句话。开发者吃下复杂度，换用户一个简单入口。这就是从「用 AI」到「造 AI 产品」的跃迁。

思考实验 · 你会怎么选？
  1. 场景 A：你想每周自动给老板发一份铁矿石库存周报。→ 选哪个？为什么？
  2. 场景 B：你接手一个完全陌生的代码库，要搞懂它干嘛的。→ 选哪个？为什么？
  3. 场景 C：你要让公司里不懂技术的销售，能自己查「这周库存预测涨还是跌」。→ 选哪个？
参考思路：A 选自建 agent（定时调用、输出稳定）；B 选 Claude Code（未知问题、要即兴探索）；C 选自建 agent（给非技术人员用、要平民化）。关键看：流程是否可预见 + 受众是否懂技术。
3 题自测
自建 agent 和 Claude Code 最根本的区别是什么？
看答案
谁在设计流程。Claude Code 把流程交给 AI（放权），自建 agent 把流程攥在人手里（约束），AI 只在决策点填智能。
「agent 是约束工程」这句话怎么理解？
看答案
搭 agent 的核心动作不是调教模型，而是设计约束（固定流程图、窄工具、固化 prompt、结构化输出）。每一层约束用灵活性换可控性，把不确定性约束掉让结果可预期。

📖 术语表

Agent（智能体）
能自己决定调哪个工具、走哪一步的 AI 程序。流程由 LLM 动态选，不是写死的。
LangGraph
把任务画成"节点+边"的图来编排的框架。节点是干活函数，边是执行顺序，state 是共享白板。
Node / Edge（节点/边）
节点 = 一个干活的函数（如 planner_node）。边 = 节点间的跳转。条件边 = 由函数运行时决定下一站。
State（状态）
所有节点共享的数据对象，像团队白板。节点读它取上游结果、写它存自己的产出。
MCP（Model Context Protocol）
AI 调用外部工具/数据的协议。给 AI 装"USB 接口"，插上工具就能用。本项目的 MCP server 是"档案室"。
FastMCP
mcp 库里快速搭 MCP server 的类，像 Flask。用 @mcp.tool() 定义工具。
streamable_http
MCP 的一种传输方式，走 HTTP。本项目的 MCP server 用它，监听 17000。
SSE（Server-Sent Events）
服务器主动推消息的协议。这里把 AI 生成过程一条条事件实时推给客户端。
ReAct
"思考-行动"循环：LLM 想一步→调工具→看结果→再想。researcher 就是 ReAct agent。
checkpointer
把 state 持久化的机制，让流程能中断/恢复。本项目用 MemorySaver（内存版，免 Postgres）。
ChatAnthropic vs ChatOpenAI
两种 LLM 客户端，分别走 Anthropic 协议和 OpenAI 协议。本项目用 ChatAnthropic 接火山方舟 GLM。
thinking 块
模型返回的"思考过程"内容。Anthropic 协议会返回，占 token，content 变列表，需提取 text。

L1 · 共享引擎（先学语法，再读句子）

这一层回答
每个节点都在调、但指南从没打开的 3 个机制是什么？
💡 一句话答案
① LLM 层（统一造/缓存模型客户端）② ReAct+中间件（造一个会调工具的 agent）③ 上下文压缩（别让对话撑爆 token）。三者分别在 src/llms/llm.py、src/agents/agents.py、src/utils/context_manager.py，被 nodes.py 所有节点复用。

L1-1 · LLM 层：get_llm_by_type + 多 vendor

所有节点不直接 ChatOpenAI(...)，而是 get_llm_by_type(llm_type) 拿一个模型客户端。它做两件事：① 缓存（同类型只造一次）② 按 vendor 分发：

# agent_service/src/llms/llm.py
def get_llm_by_type(llm_type):
    if llm_type in _llm_cache: return _llm_cache[llm_type]   # 全局缓存
    llm = _create_llm_use_conf(llm_type)
    _llm_cache[llm_type] = llm
    return llm

# _create_llm_use_conf 按 LLM_VENDOR 造不同客户端：
#   hiagent    -> CustomChatOpenAI（extra_body 注入 enable_thinking）
#   anthropic  -> ChatAnthropic（火山方舟 GLM 走这）  ← Q6 Bug1 的 content 列表来自这
#   doubao/qwen/deepseek/deepseekv4 -> 各自的 extra_body
回扣 Q6 Bug 1

指南 Q6 说"ChatAnthropic 返回的 content 是列表"。但为什么用 ChatAnthropic？因为 LLM_VENDOR=anthropic 时这层选了它。指南修了症状（_content_to_text），根因在这层的 vendor 分发。读懂这层，你就懂"换模型只改 env 不改代码"是怎么做到的。

L1-2 · ReAct + 中间件：create_agent

指南 Q3/Q4 说"researcher 是 ReAct agent"，但怎么造的从没讲。答案在 create_agent（src/agents/agents.py）：底层调 langchain.agents.create_react_agent，关键是挂了 3 个中间件：

# agent_service/src/agents/agents.py · create_agent
middleware = [
    SummarizationMiddleware(model=..., trigger=("tokens", 20000), keep=("tokens", 10000), ...),  # 超 2 万 token 自动摘要历史
    ToolCallLimitMiddleware(run_limit=5),   # 一个 agent 单次最多调 5 次工具
    CaptureToolCallIdMiddleware(),          # 抓 tool_call id，给 SSE 事件匹配用
]
agent = create_react_agent(name=graph_name, model=get_llm_by_type(llm_type),
                           tools=tools, middleware=middleware,
                           system_prompt=render_system_prompt(...))
中间件 = agent 的"行为政策"

ReAct 循环本身（想→调工具→看结果→再想）是 create_react_agent 内置的；中间件决定它什么时候停、什么时候压缩。ToolCallLimitMiddleware(5) 就是为什么 researcher 调几个工具后就收手——不是 LLM 自己决定的，是中间件拦的。SummarizationMiddleware 是第二套上下文压缩（见 L1-3）。

L1-3 · 上下文压缩：ContextManager + SummarizationMiddleware

指南 Q2 的 planner 代码里有个 compressed_state 从没解释——就是它。项目有两套压缩：

① ContextManager（节点入口，手写）

每个节点开头调：compressed_state = ContextManager(token_limit, 保留前N条).compress_messages(state)。逻辑：估算 token（中文1字≈1 token，英文4字≈1 token）→ 超限就保留前 N 条 + 从尾部往回留 → 实在放不下就截断单条。

nodes.py 里 planner(449)、coordinator(725)、reporter(1147)、researcher(1286)、tool_executor(1641) 全调它。

② SummarizationMiddleware（ReAct 内部，自动）

只在 researcher 这类 ReAct agent 的多轮工具循环里生效：对话攒到 20000 token 触发，用 LLM 把旧消息摘要成一段、保留最近 10000 token。

一个管"节点入口的 state 太长"，一个管"agent 内部循环越滚越长"。

动手实验 L1 · 摸中间件的边界
  1. 打开 .env.dev，加一行 AGENT_REACT_TOOLS_LIMIT=2。
  2. 重启 agent_service（终端 2），终端 3 重跑报告。
  3. 盯日志：researcher 调 2 次工具后会被强制停下，出现"超出工具调用限制"。
  4. 改回 5（或删掉这行），重启确认恢复。
你会观察到：agent 调几次工具不是 LLM 说了算，是 ToolCallLimitMiddleware 拦的。这就是"中间件=行为政策"。
