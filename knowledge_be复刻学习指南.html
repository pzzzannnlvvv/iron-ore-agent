Q1 · 一篇文档怎么变成能被「搜索」的东西?

驱动问题
我往 data/corpus 丢一堆 md/html，系统怎么把它变成可秒查的形态？为什么要先「切」再「编码」？
💡 一句话答案
入库管道四步：load（抽纯文本）-> split（切成 chunk）-> embed（每个 chunk 编成 1024 维向量）-> store（存进 Chroma）。15 篇文档最终变成 237 个带向量的 chunk，这就是「可搜索」的形态。

为什么不直接存整篇文档？要先切

一篇报告可能几千字，但你要找的可能只是其中「MDA 84.91%」那一句。如果整篇当一个单位编码、检索，返回的是整篇文档，精度差、上下文太长喂不进 LLM。所以切成几百字一块（chunk），检索时返回的就是精准的小片段。

# knowledge_service/src/pipeline/ingest.py · ingest()
store.reset()                              # 清空旧库
docs = load_corpus(corpus_dir)             # ① 读 15 篇文档
for doc in docs:
    chunks = split_text(doc["content"], CHUNK_SIZE, CHUNK_OVERLAP)  # ② 切块
    embs = embed_batch(chunks)             # ③ 每块编码成向量
    for i, (chunk, emb) in enumerate(zip(chunks, embs)):
        batch.append({
            "id": f"{doc['doc_id']}_{i:04d}",      # doc_003_0017 这种 id
            "content": chunk,
            "embedding": emb,
            "metadata": {"fileName": doc["file_name"], "sourceType": 1, ...},
        })
    store.add_chunks(batch)                # ④ 存进 Chroma
bm25.reset_index()                         # 顺带重建 BM25 索引

split 怎么切：递归 + 中文友好

不是无脑按 500 字硬切（会把句子拦腰斩断）。它按 段落 -> 换行 -> 句号 -> 分号 -> 逗号 逐级切，先大后小，切到 <= 500 字就停，再贪心合并相邻片段成 chunk，还带 50 字 overlap（重叠，保证上下文不断裂）：

# knowledge_service/src/pipeline/splitter.py
SEPARATORS = ["\n\n", "\n", "。", "!", "?", ";", ..., "，", " ", ""]
# 先按段落切，某段太长就按句号切，还太长按分号... 最后兜底硬切

cur += p
if len(cur) > chunk_size:
    chunks.append(cur)
    tail = cur[-chunk_overlap:]   # 留 50 字尾巴
    cur = tail + p                # 下一段开头接上尾巴，防断裂
为什么留 overlap

假设一个结论正好被切在两块边界：「...MDA 方向准确率达到」|「84.91%，优于基线...」。没 overlap 的话，检索「MDA 多少」可能两块都半截。留 50 字重叠，前一块带「84.91%」，命中后上下文更完整。

3 题自测（点开看答案）
为什么不把整篇文档当检索单位，非要切片？
看答案
两个原因：① 精度——你要的是「MDA 84.91%」那一句，返回整篇几千字没法用；② 长度——检索结果要喂给 LLM，整篇塞不下 prompt，也太贵。切块让返回的又准又短。
splitter 的 SEPARATORS 为什么要排这么多、有先后顺序？
看答案
按「语义边界」从大到小切：先段落（最自然边界）-> 换行 -> 句号 -> 分号 -> 逗号。优先在大边界切，切不下才退到小边界，最后才硬切。这样 chunk 尽量不断在句子中间，语义完整。
chunk_overlap=50 在解决什么问题？
看答案
解决「结论被切在两块边界」的断裂问题。每块尾部留 50 字重叠到下一块开头，保证横跨边界的信息至少在一块里完整出现。代价是少量冗余存储。

Q2 · bge-m3 到底是什么？它和 LLM（glm）是一回事吗？

驱动问题
我桌面上那个 bge-m3（2.27GB）是大模型吗？能不能像 glm 那样对话生成文字？为什么要专门下它，不能用我已有的 glm 配置替换？
💡 一句话答案
不是 LLM，不能对话。bge-m3 是 embedding 双编码器，只干一件事：把一段文本压成一个 1024 维向量（语义指纹）。它不生成文字。glm 是生成式 LLM。两者是完全不同物种的模型，glm 替代不了它。

两种模型，两种用途

glm（生成式 LLM）

输入一段文本，输出一段新生成的文本。

会推理、会写、会对话。

慢、贵、不可预测输出。

用在 agent 里做规划/研究/写报告。

生成式

bge-m3（embedding 模型）

输入一段文本，输出一个固定 1024 维向量。

不会生成任何文字，只「编码」。

快、便宜、输出确定。

用在知识库里把文本变成可比较的向量。

编码式

为什么不能用 glm 替代 bge-m3？因为算向量相似度需要一个确定、可比的数值表示。bge-m3 把「铁矿石库存下降」和「港口铁矿库存减少」编成两个非常接近的向量（语义近），把「今天天气不错」编成很远的向量。glm 做不到这件事——它只会接话，不会把句子变成坐标。

embed 怎么用 bge-m3

# knowledge_service/src/pipeline/embedder.py
from FlagEmbedding import BGEM3FlagModel

@lru_cache(maxsize=1)          # 进程内只加载一次（2.27GB，不能每次查都重载）
def get_model():
    path = os.getenv("BGE_M3_PATH", DEFAULT_BGE_M3_PATH)
    use_fp16 = os.getenv("BGE_USE_FP16", "0") == "1"  # 无 GPU，fp32
    return BGEM3FlagModel(path, use_fp16=use_fp16)

def embed(text, normalize=True) -> list[float]:
    model = get_model()
    out = model.encode([text], return_dense=True, return_sparse=False, ...)
    vec = np.asarray(out["dense_vecs"][0], dtype=np.float32)
    if normalize:                    # 归一化：让向量长度=1，cosine 就退化成点积
        vec = vec / np.linalg.norm(vec)
    return vec.tolist()             # 1024 维 list
两个关键设计

① lru_cache 懒加载：模型 2.27GB，加载要几秒+占内存。get_model() 第一次调用才加载，之后全程复用同一个实例。这就是为什么服务启动后第一次检索慢、之后飞快。② normalize 归一化：把向量缩成长度 1，这样两个向量的余弦相似度 = 简单的点积，Chroma 算得快。

3 题自测
bge-m3 能像 glm 那样回答问题吗？为什么？
看答案
不能。bge-m3 是编码器，输入文本输出向量，没有「生成下一个 token」的解码头。它只把文本变成坐标，不产生新文字。glm 有解码器能生成。
为什么 get_model() 要套 lru_cache？
看答案
bge-m3 2.27GB，加载耗时几秒+占大量内存。每次检索都重载会慢且爆内存。lru_cache 让模型进程内只加载一次、全程复用。所以首次检索慢、之后快。
为什么要 normalize 归一化向量？
看答案
归一化后向量长度=1，两向量的余弦相似度就等于点积（一个乘加运算），计算极快。Chroma 用 cosine 空间，归一化让相似度计算又快又准。

Q3 · 光靠「向量相似度」搜索够吗？为什么还要 BM25？

驱动问题
既然 bge-m3 向量能抓语义，为什么 search.py 不只跑 KNN 向量召回，还要再跑一遍 BM25 关键词召回再融合？多此一举？
💡 一句话答案
因为向量检索和关键词检索各有盲区。向量抓语义但会漏精确专有名词（如「MDA」「LGBM」）；BM25 抓词面匹配但不懂同义。两路互补 + RRF 融合，比任何单路都准。这就是「混合检索」。

两路召回各自的盲区

只有 KNN（向量）

  • 懂语义：「库存减少」能查到「库存下降」
  • 盲区：精确术语常被「稀释」。查「LightGBM」可能因为语义接近「模型」被泛泛文档挤掉
  • 对罕见专有名词不敏感

只有 BM25（关键词）

  • 精确匹配：「LightGBM」三个字母必须出现才命中
  • 盲区：不懂同义。查「库存下降」查不到只写「库存减少」的文档
  • 对拼写/表述变化无能为力

混合检索：两路都跑，各出一份排名，再用 RRF 融合成一份。向量补 BM25 的「不懂同义」，BM25 补向量的「漏专有名词」。

BM25 怎么工作：jieba 分词 + 词频统计

# knowledge_service/src/retrieval/bm25.py
import jieba
from rank_bm25 import BM25Okapi

def _tokenize(text, for_search=False):
    tokens = jieba.cut_for_search(text) if for_search else jieba.cut(text)
    return [w for w in tokens if w.strip()]   # 中文要先分词！

# 建索引：把 237 个 chunk 全分词，统计词频
self.tokenized = [_tokenize(d) for d in self.docs]
self.bm25 = BM25Okapi(self.tokenized)

# 查询：query 分词后对每个 chunk 打分
scores = self.bm25.get_scores(_tokenize(query, for_search=True))
中文 BM25 必须先分词

英文 BM25 按空格切词就行，中文不行——「铁矿石库存」按空格切是整一坨。必须用 jieba 先切成「铁矿石/库存」这种词。查询用 cut_for_search（更细，召回更多），建索引用 cut（标准切分）。这是中文 RAG 的标配。

RRF：怎么把两份排名合成一份

KNN 给一份按向量距离的排名，BM25 给一份按词频的排名，两份排名分数不可比（一个是 0~1 余弦，一个是 BM25 绝对分）。RRF 的聪明之处：不看分数，只看排名。

# knowledge_service/src/retrieval/hybrid.py
def rrf(rankings, k=60):
    scores = {}
    for ranking in rankings:            # [vec_hits, bm_hits] 两路
        for rank, hit in enumerate(ranking):   # rank 从 0 开始
            hid = hit["id"]
            scores[hid] = scores.get(hid, 0) + 1.0 / (k + rank + 1)   # 排名越靠前分越高
    # 一个 chunk 两路都命中 -> 分数叠加，自然排到前面
    return sorted(scores.items(), key=lambda x: -x[1])

公式 1/(k+rank+1)，k=60 是个平滑常数。关键效果：两路都召回的 chunk 双倍加分，自然冒泡到最前面——这种「共识」chunk 往往就是最相关的。

3 题自测
为什么 KNN 和 BM25 的分数不能直接相加，要用 RRF？
看答案
两路分数不在一个量纲：KNN 是 0~1 的余弦相似度，BM25 是无上界的词频分。直接加会被 BM25 大数主导。RRF 只用排名（1/(k+rank)），抛弃分数量纲，让两路公平贡献。
中文做 BM25 为什么必须先 jieba 分词？
看答案
中文没有空格，「铁矿石库存」是一整串不分词就是一个 token，匹配不到只含「库存」的文档。jieba 切成「铁矿石/库存」后才能按词统计词频、做匹配。英文按空格切即可，中文必须分词。
一个 chunk 被 KNN 和 BM25 两路都召回，RRF 会怎么对它？
看答案
两路的排名分都加到它头上，分数叠加，自然排到融合结果最前面。这种「双路共识」chunk 通常就是最相关的，这正是 RRF 的核心效果。

Q4 · 召回一堆候选了，为什么还要再「重排」一遍？

驱动问题
BM25+KNN+RRF 已经排好序了，search.py 最后又调 bge-reranker 重新打分重排。这不是重复劳动吗？
💡 一句话答案
不是重复，是精度升级。第一路召回用的是双编码器（bi-encoder，query 和 doc 各自独立编码再算距离，快但浅）；reranker 是交叉编码器（cross-encoder，query+doc 拼一起喂进模型，理解深但慢）。两段式：粗筛（快）-> 精排（准）。

bi-encoder vs cross-encoder：召回 vs 重排的本质区别

bi-encoder（bge-m3，用于召回）

  • query 和 doc 各自编码成向量，再算距离
  • doc 可预编码存库，查询只编 query -> 超快
  • query 和 doc 没互相看到，理解浅
  • 适合从 237 个 chunk 里粗筛几十个

cross-encoder（bge-reranker，用于重排）

  • query 和 doc 拼一起喂进模型，一起编码
  • 每对 (query,doc) 都要跑一次模型 -> 慢
  • 模型能看 query 和 doc 的交互关系，理解深
  • 适合对几十个候选精排

为什么不能直接用 cross-encoder 检索全部？因为 237 个 chunk × 每次 query 都要跑一遍 cross-encoder = 太慢。所以先用快的 bi-encoder 粗筛出 top30，再用慢但准的 cross-encoder 精排 top30。这就是「两段式检索」，RAG 的标准操作。

rerank 怎么用 bge-reranker

# knowledge_service/src/pipeline/reranker.py
from transformers import AutoModelForSequenceClassification, AutoTokenizer

@lru_cache(maxsize=1)
def get_model():
    tokenizer = AutoTokenizer.from_pretrained(path)
    model = AutoModelForSequenceClassification.from_pretrained(path)  # XLMRoberta 分类头
    model.eval()
    return tokenizer, model

def rerank(query, documents, normalize=True, max_length=512):
    pairs = [[query, doc] for doc in documents]   # 关键：query+doc 拼成 pair
    inputs = tokenizer(pairs, padding=True, truncation=True, ...)
    with torch.no_grad():
        logits = model(**inputs).logits.squeeze(-1)        # 每对一个分
    scores = torch.sigmoid(logits) if normalize else logits   # sigmoid -> 0~1
    return scores.tolist()
cross-encoder 的关键在「pair」

看那行 pairs = [[query, doc] ...]——query 和 doc 被拼成一对一起进模型。模型内部 attention 让 query 的每个词都能「看到」doc 的每个词，捕捉「这半句到底在回答 query 的哪个点」。bi-encoder 做不到这种交互，因为 query 和 doc 是分开编码的。这就是精排更准的根本原因。

search.py 怎么接 rerank

# search.py
candidates = merged[: max(topK * 2, 20)]     # 融合后取前 20 个候选
if reRank == 1 and candidates:

3 题自测
为什么不能一开始就用 cross-encoder 检索全部 237 个 chunk？
看答案
太慢。cross-encoder 要对每个 (query, chunk) 都跑一次完整模型（237 次），且无法预编码。bi-encoder 能预编码存库、查询只编 query，秒级出结果。所以先用 bi 粗筛再用 cross 精排。
bi-encoder 和 cross-encoder 在「query 和 doc 怎么进模型」上的根本区别？
看答案
bi-encoder 把 query 和 doc 各自独立编码成向量再算距离，两者没交互；cross-encoder 把 query 和 doc 拼成 pair 一起进模型，attention 让两者词词交互，能捕捉「doc 哪部分在回答 query」。后者理解深但慢。
reranker 输出的分数为什么过 sigmoid？
看答案
模型输出的是 logit（无界实数），sigmoid 压到 (0,1) 区间，变成可解释的「相关概率」。越大越相关，且跨 query 可比。也方便排序和过滤。

Q5 · 检索中途崩了怎么办？系统怎么不挂？

驱动问题
如果 bge-m3 模型加载失败、内存爆了，或者 reranker 抽风，整个 search 是不是就废了、用户拿不到任何结果？
💡 一句话答案
不会。search.py 设计了两层降级：① embedding 失败 -> 自动退化为纯 BM25（还能出结果）② rerank 失败 -> 用 RRF 融合分排序（结果还在，只是没那么准）。服务永远尽量不空手而归。

降级 1：embedding 失败 -> 纯 BM25

# search.py
t0 = time.time()
emb_ok = True
qvec = None
try:
    qvec = embed(content)              # bge-m3 编码 query
except Exception as e:
    log.warning("embedding 失败,降级纯 BM25: %s", e)
    emb_ok = False                       # 标记降级

# 召回阶段：
bm_hits = bm25.get_index().search(content, top_k=pool_k)
if not emb_ok or mode == "bm25":
    merged = bm_hits                   # 降级：只用 BM25，跳过向量
elif mode == "knn":
    merged = vector.search_vec(qvec, top_k=pool_k)
else:  # mix
    vec_hits = vector.search_vec(qvec, top_k=pool_k)
    merged = hybrid.rrf([vec_hits, bm_hits])

即使 bge-m3 完全挂掉，BM25 不依赖模型（只靠 jieba+词频），照样能按关键词召回。用户还能拿到结果，只是丢了语义检索能力。

降级 2：rerank 失败 -> 用融合分

# search.py
if reRank == 1 and candidates:
    try:
        scores = rerank_fn(content, docs)
        for c, s in zip(candidates, scores): c["score"] = float(s)
        candidates.sort(key=lambda x: -x["score"])
    except Exception as e:
        log.warning("rerank 失败,用融合分排序: %s", e)   # 不重排，保留 RRF 分
candidates = candidates[:topK]

reranker 挂了也不致命：候选还在，只是按 RRF 融合分排（粗一点），不按精排分。用户拿到的还是 topK 结果。

这是工程上的「优雅降级」（graceful degradation）

核心思想：核心功能挂了，用次优方案顶上，而不是整体崩。向量挂了->关键词顶；重排挂了->融合分顶。每层都有 fallback。这和原版 knowledge_be 的降级逻辑一致（复刻时专门对齐的）。真实生产 RAG 必须这样——模型总会抽风，不能让一次抽风让整个服务返回 500。

3 题自测
为什么 embedding 失败后还能用 BM25 出结果？
看答案
因为 BM25 完全不依赖 embedding 模型——它只靠 jieba 分词 + 词频统计，是纯算法。bge-m3 挂了不影响 BM25。降级就是把向量路跳过、只走 BM25 路。
rerank 失败时，候选的排序依据是什么？
看答案
用 RRF 融合分（mix 模式下 KNN+BM25 融合的排名分）。candidates 已经按融合分排好了，rerank 失败就不覆盖这个分、不重排，直接裁 topK。结果还在，只是精度降一档。
「优雅降级」对生产 RAG 为什么重要？
看答案
模型推理总会偶发失败（OOM、加载慢、权重损坏）。如果没有降级，一次抽风就让整个 search 返回 500、用户拿不到任何东西。优雅降级让核心挂了用次优顶上，服务可用性大幅提升。每层都设 fallback 是工程标配。

Q6 · 联调时 reranker 直接报错，怎么像侦探一样定位修复？

驱动问题
第一次跑 reranker 烟测直接 AttributeError 崩了。真实项目里这种「库能用但跑起来报错」的 bug 怎么找、怎么修？
💡 一句话答案
报错栈指到 FlagReranker 调用了 tokenizer.prepare_for_model，而 transformers 5.x 删了这个方法。根因=库版本不兼容。修复=绕开 FlagReranker，直接用 AutoModelForSequenceClassification + AutoTokenizer 标准 cross-encoder 写法（跨版本稳定）。

Bug 现场

报错栈
AttributeError: 'XLMRobertaTokenizer' object has no attribute 'prepare_for_model'
  File ".../FlagEmbedding/flag_reranker.py", line XX, in compute_score
    inputs = self.tokenizer.prepare_for_model(...)
                 ↑
            transformers 5.x 已删除这个方法

侦探步骤 1 · 读栈定位：报错不是出现在我们的代码，而是 FlagEmbedding/flag_reranker.py 内部。说明我们的调用方式没错，是 FlagEmbedding 这个库自己崩了。

侦探步骤 2 · 抠关键词：has no attribute 'prepare_for_model'——FlagReranker 想调 tokenizer.prepare_for_model，但这个 tokenizer 对象没有这方法。

侦探步骤 3 · 查根因：为什么没有？因为我们装的 transformers 5.x 把 prepare_for_model 这个老方法删了（5.x 大版本清理了一批废弃 API）。而 FlagEmbedding 1.4.0 的 FlagReranker 还在用它。于是 库版本不兼容。

根因

FlagEmbedding 的 FlagReranker 依赖 tokenizer.prepare_for_model，但 transformers 5.x 删了这方法。两个库的版本对不上。这是 Python 生态最常见的「依赖地雷」——A 库能用，B 库能用，但 A+B 的特定版本组合会炸。

为什么 embedder 没崩、只有 reranker 崩？

有趣的细节：embedder 用的 BGEM3FlagModel（同一个 FlagEmbedding 库）没崩，只有 FlagReranker 崩。因为两者走不同的 tokenizer 代码路径——BGEM3FlagModel 没碰 prepare_for_model，FlagReranker 碰了。同一个库里，不同模块对底层 API 的依赖不同，兼容性也不同。

修复：绕开 FlagReranker，用标准写法

修复前（用 FlagReranker）

from FlagEmbedding import FlagReranker
reranker = FlagReranker(path)
scores = reranker.compute_score(pairs)
# 内部调 prepare_for_model -> 崩

修复后（直接用 transformers）

tokenizer = AutoTokenizer.from_pretrained(path)
model = AutoModelForSequenceClassification.from_pretrained(path)
inputs = tokenizer(pairs, padding=True, ...)
logits = model(**inputs).logits
# 不碰 prepare_for_model,跨版本稳

这是标准的 cross-encoder 写法，不经过 FlagEmbedding 的封装，直接用 transformers 原生 API。FlagReranker 本质也就是这么做的，只是它额外包了一层恰好踩了雷的 prepare_for_model。我们绕开那层，自己写等价逻辑，跨版本稳定。

学到什么

① 报错栈要顺藤摸到根因，别只看「报错了」就慌。这次根因不在我们代码、在第三方库内部。② 「库能用但报错」常是版本不兼容，抠出缺失的 attribute 名去搜，基本能定位。③ 能绕开就绕开：库的封装崩了，不一定非要降级库版本（可能引发别的依赖问题），用底层 API 自己写等价逻辑往往更稳。④ 同一个库不同模块兼容性不同，embedder 没崩不代表 reranker 不会崩。

凭什么算「复刻」？——语义对齐是关键

复刻成立的根：用 bge 同款模型

复刻 knowledge_be，判断「要对齐」和「可简化」的标准是什么？
看答案

📖 术语表

RAG（检索增强生成）
Retrieval-Augmented Generation。先从知识库检索相关片段，再喂给 LLM 生成答案。本指南做的是「R」（检索）那半。
Embedding（向量化/嵌入）
把文本压成定长向量（bge-m3 是 1024 维）。语义相近的文本向量也相近。是向量检索的基础。
bi-encoder（双编码器）
query 和 doc 各自独立编码成向量再算距离。可预编码、快、但理解浅。bge-m3 是这类，用于召回。
cross-encoder（交叉编码器）
query 和 doc 拼一起进模型打分。能捕捉两者交互、理解深，但慢。bge-reranker 是这类，用于重排。
dense / sparse 向量
dense=稠密向量（每维都有值，bge-m3 产出）；sparse=稀疏向量（多数维为 0，BM25 词频可视为 sparse）。向量检索用 dense，关键词检索本质是 sparse。
cosine 相似度
两向量夹角的余弦，范围 -1~1。越大越相似。归一化后=点积。Chroma 用 cosine 空间。
BM25
基于词频的经典关键词检索算法。词在文档出现越多、在全局越稀有，分越高。中文要先 jieba 分词。
jieba
中文分词库。把「铁矿石库存」切成「铁矿石/库存」。中文 BM25 必备。
RRF（Reciprocal Rank Fusion）
多路召回的融合算法。只用排名不算分数：score = Σ 1/(k+rank)。让多路共识的 chunk 冒泡到前面。
chunk / chunking
把长文档切成几百字的小块。检索单位是 chunk 不是整篇。chunk_size/overlap 是关键参数。
Chroma
轻量向量数据库，本地文件持久化。本复刻用它替代原版的 ES，零运维。
normalize（归一化）
把向量缩成长度 1。归一化后 cosine 相似度=点积，计算快。
FP16
半精度浮点。GPU 推理用 FP16 省显存加速。本机无 GPU，用 FP32（BGE_USE_FP16=0）。
sentence-transformers
加载 sentence embedding 模型的标准格式。bge-m3 兼容此格式。
KnowledgeSearchVO
原版 knowledge_be 的检索结果数据结构（id/content/fileName/source/score 等字段）。复刻用 _to_vo 对齐。
降级（degradation）
核心功能失败时用次优方案顶上。embed 挂->纯 BM25；rerank 挂->用融合分。保证服务不空手而归。
pool_k（候选池）
召回阶段多取一些（max(topK*3,30)）作为 rerank 候选，再裁到 topK。粗筛留余量给精排。
两段式检索
先用快的 bi-encoder 粗筛几十个，再用慢的 cross-encoder 精排。兼顾速度和精度，RAG 标准操作。

L1-3 · 模型懒加载缓存：lru_cache

embedder 和 reranker 的 get_model() 都套 @lru_cache(maxsize=1)。这是整个服务性能的关键：

@lru_cache(maxsize=1)          # 进程内只执行一次
def get_model():
    return BGEM3FlagModel(path, ...)   # 2.27GB，加载几秒
为什么这是性能关键

没有 lru_cache，每次检索都重载 2.27GB 模型，每次都要几秒+占双倍内存。有了它，模型加载一次常驻内存，之后每次检索只是 model.encode() 推理。这就是为什么服务启动后首次检索慢、之后飞快。store.py 的 _collection 单例、bm25 的 _index 单例也是同理。

L3-1 · 过滤：_filter 日期/来源后处理

search.py 有个 _filter 函数，在融合后、rerank 前做 发布日期 和 来源类型 过滤：

# search.py · _filter
for h in hits:
    pt = md.get("publishTime")
    if publish_start and pt and pt < publish_start: continue   # 早于起始
    if publish_end and pt and pt > publish_end: continue     # 晚于结束
    if source_type is not None and md.get("sourceType") != source_type: continue
为什么在 Python 侧后处理，不用 Chroma 过滤？

因为 Chroma 的 metadata 过滤能力较弱（日期比较不灵活），所以召回时不过滤、拿回来后在 Python 侧按 publishTime/sourceType 筛。这是「召回多取 -> 后处理精筛」的常见模式。
