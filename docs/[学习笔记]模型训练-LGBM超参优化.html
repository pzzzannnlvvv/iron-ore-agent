🎯 一、一句话：我们要做什么？

用 Optuna（一个自动调参工具）帮 LightGBM（一个预测模型）找到最优的参数设置，
让模型预测铁矿石库存的误差尽可能小。
最终只保存参数组合（不保存几百 MB 的模型文件），
并提供一套可 100% 复现结果的脚本。
简单比喻：这就像你想做一道最好吃的菜（预测模型），但盐放多少、火候多大、煮多久（超参数）需要反复尝试。Optuna 就是一个自动帮你尝味道、调整配方的大厨助手，它比你自己一次次试（网格搜索）快得多、聪明得多。

四个核心目标

#目标大白话解释
1用 Optuna 替代网格搜索不让机器盲目遍历所有组合，而是「聪明地猜」哪些参数更值得试
2只保存超参+指标，不保存模型文件模型文件很大（几百 MB），参数只是几十行数字，用 CSV 表格存就够了
3推理脚本 100% 复现结果给定同样的参数和同样的数据，运行两次得到完全一样的结果
4交付 2 个脚本 + 1 个结果文件优化脚本、复现脚本、参数结果表（CSV），三个文件即可

📖 二、核心概念辞典（必读）

先搞清楚这些词是什么意思，后面的内容才能看懂。每个概念都附了「大白话类比」。

超参数 Hyperparameter
模型训练前由人设定的参数，模型自己学不会。比如「树最多长多深」「每次学多快」。 比喻：做菜时「盐放多少」「火开多大」——不是菜自己决定的，是厨师调的。
超参优化 / 调参 Hyperparameter Optimization
自动搜索最优超参数组合的过程。目标：找到让模型预测误差最小的那组参数。 比喻：反复做同一道菜，每次微调配方，直到找到最好吃的那个版本。
网格搜索 Grid Search
把所有参数组合列成一张大表，一个一个试。比如 5 个参数各取 10 个值 = 10⁵ = 10 万次试验，非常慢。 比喻：想找到从北京到上海的最优路线，于是把全国每条路都走一遍——太慢了。
Optuna
一个日本公司 Preferred Networks 开源的自动超参优化框架。核心思想：根据已经试过的参数效果，智能推测下一组该试什么参数，比网格搜索快 10~100 倍。 比喻：一个聪明的厨师学徒——尝过几次后就知道往哪个方向调整配方，不需要把每种组合都做一遍。
TPE 采样器 Tree-structured Parzen Estimator
Optuna 默认使用的智能搜索算法。它会根据历史试验结果，把「好参数」和「差参数」分成两组，然后优先从「好参数」区域采样新的尝试值。 比喻：你在调台灯的亮度，发现太亮和太暗都不舒服，TPE 会自动聚焦到让你舒服的亮度范围去试。
剪枝 / MedianPruner Pruning
训练过程中，如果某组参数的中间结果已经明显很差（低于所有已完成试验的中位数），就提前终止这组试验，不浪费算力。 比喻：做菜时尝了一口发现咸得没法吃，直接倒掉重做，而不是做完再倒。
交叉验证 Cross Validation, CV
把训练数据切成 N 份，轮流用 N-1 份训练、1 份验证，取 N 次结果的平均值作为最终评分。避免「碰巧在某一份数据上表现好」的假象。 比喻：模拟考不能只做一套卷子，要做多套不同卷子取平均分，才能真实反映水平。
时序交叉验证 TimeSeriesSplit
专门给时间序列数据用的交叉验证。和普通 CV 的区别：不能用未来数据验证过去。每次切分时，训练数据的时间一定在验证数据之前。 比喻：不能用明天的股价来验证今天的预测模型——那叫作弊（数据泄露）。
随机种子 Random Seed
计算机的「随机」其实是伪随机——由一个初始数字（种子）决定整个随机序列。同样种子 → 同样随机序列 → 同样结果。 比喻：Minecraft 的地图种子——输入同一个种子号，生成的世界一模一样。
DataFrame
Python 的 pandas 库中最核心的数据结构，可以理解为一个「超级 Excel 表格」，有行有列，可以筛选、排序、计算、导出 CSV。
梯度提升树 Gradient Boosting Decision Tree, GBDT
LightGBM 的核心算法。简单说：先建一棵简单的树做预测，然后第二棵树专门学习第一棵树的「残差」（预测值和真实值的差距），第三棵树学习前两棵的残差……这样一棵一棵叠加，预测越来越准。 比喻：团队合作解题——第一个人给出初步答案，第二个人专门修正前面的错误，第三个人再修正剩下的错误，越改越准。
过拟合 Overfitting
模型在训练数据上表现极好，但在新数据（测试集）上表现很差。相当于「死记硬背了答案，但换个题就不会做」。 比喻：学生把练习册答案全背下来了，考试满分；但换一套新题就不会了。
正则化 Regularization
防止过拟合的手段。通过在损失函数中加入惩罚项，限制模型参数不要太大或太复杂。L1（Lasso）会直接把不重要的参数压成 0；L2（Ridge）会让所有参数都尽量小。 比喻：给模型套上缰绳，不让它跑太偏。

🌲 三、LightGBM 是什么？

LightGBM（Light Gradient Boosting Machine）是微软 2017 年开源的一个梯度提升决策树框架。 它是 XGBoost 的「升级版」，核心优势是训练速度快、内存占用少、精度高，在 Kaggle 竞赛和工业界使用极广。

它是怎么工作的？（三步理解）

1
初始化：先用一个常数（比如所有目标值的平均值）作为初始预测。比如预测铁矿石库存，第一轮全部猜「平均库存量」。
2
迭代修正：每轮新建一棵决策树，专门学习上一轮预测的「残差」（真实值 - 预测值）。新树的目标就是「把上一轮没预测准的部分补上」。
3
累加输出：最终的预测 = 初始值 + 第1棵树的修正 + 第2棵树的修正 + …… + 第N棵树的修正。所有树的结果累加。
LightGBM vs XGBoost 的核心区别：LightGBM 使用 直方图算法（Histogram）和 按叶子生长（Leaf-wise）策略，而 XGBoost 按层生长（Level-wise）。这使得 LightGBM 更快、更省内存，但更容易过拟合——所以更需要认真调参。

🔬 三-B、LightGBM 核心训练逻辑详解

深入理解树是怎么生长的、残差是怎么存储的、随机种子到底控制什么。

一、核心训练逻辑（梯度提升思路）

逐树叠加、拟合残差
1
第一棵树：用一个常数（比如所有目标值的平均值）做基础预测
2
后续每棵树：专门学习前面所有树产生的残差（真实值 - 累计预测值）
3
最终预测：所有树的输出累加，模型精度逐步提升
残差是什么？预测值和真实值之间的误差，是后续每一棵树的训练目标。

二、残差在树结构中的表达方式

树不会直接存储残差数字，而是通过节点分裂 + 叶子节点取值两步共同表达残差规律：

步骤做什么作用
节点分裂 按照特征规则对样本分组 把残差相近的样本划分到同一个节点
叶子节点 无法再分裂的末端节点 节点内存储本组样本残差的均值

三、随机种子到底控制什么？

先纠正关键误区：随机种子不控制节点数量、树的大小！
什么控制树结构？什么控制随机选择？
超参数（结构参数）
max_depth、num_leaves、min_child_samples、正则化等 		随机种子
选哪个特征、选哪个分裂切点等
随机种子控制的随机行为：
  • 训练时随机选取部分特征
  • 节点分裂时随机筛选候选分裂点
  • 数据采样（如果 subsample < 1.0）
  • 特征采样（如果 colsample_bytree < 1.0）

四、整体逻辑串联（一条完整训练链路）

1. 划分数据集（随机种子固定划分结果）→
2. 初始化第一棵树，根节点包含全部样本 →
3. 树不断节点分裂：
    • 随机种子固定「选哪个特征/分裂点」
    • 结构参数+正则化限制「分裂次数、节点总数」→
4. 生成叶子节点，用叶子值表达残差，完成单棵树训练 →
5. 计算整体误差（原始损失），叠加正则惩罚，优化总损失 →
6. 计算新残差，训练下一棵树重复上述流程 →
7. 所有树输出累加，得到最终预测结果

🔧 四、什么是「超参数」和「超参优化」？

模型参数 vs 超参数

类型谁来定？举例存在哪里？
模型参数 模型自己从数据中学 每棵树的分裂点、叶子节点的预测值 模型文件（.pkl / .txt）
超参数 人（或 Optuna）在训练前设定 学习率、树深度、叶子数 一个几十行的配置 / CSV 表格

为什么需要超参优化？

  • 不同的超参组合会让同一个模型的预测精度天差地别
  • 手工试参靠经验、靠运气，效率极低
  • Optuna 能自动探索参数空间，比网格搜索快 10~100 倍
  • 它能「学习」哪些参数区域效果更好，越来越聪明地选择下一组尝试值

⚙️ 五、Optuna 是怎么工作的？

Optuna 的三板斧

① 定义目标函数（Objective Function）
你写一个函数，输入是一组超参数，输出是模型在验证集上的误差（如 MDA 或 MAPE）。Optuna 的目标就是让这个误差越小越好（MAPE 越小越好）或越大越好（MDA 越大越好，需取负数转换）。
② 采样器（Sampler）决定下一组试什么参数
我们使用 TPESampler（Tree-structured Parzen Estimator）。它维护两个概率分布：
  • 好参数分布 l(x)：历史试验中效果好的参数长什么样
  • 差参数分布 g(x)：历史试验中效果差的参数长什么样
每次选新参数时，选那些在 l(x) 中概率高、在 g(x) 中概率低的值。越试越准。
③ 剪枝器（Pruner）提前终止差的试验
我们使用 MedianPruner：在训练的每个中间步骤，如果当前试验的效果差于所有已完成试验的中位数，就直接终止。节省大量时间。

完整优化流程图

① Optuna 建议一组超参
↓
② 用这组超参训练 LightGBM（在时序交叉验证上评估）
↓
③ MedianPruner 检查：中期效果是否太差？→ 是 → 终止，回到①
↓ 否
④ 训练完成，记录 MDA / MAPE
↓
⑤ TPE 学习这组结果，更新「好参数」和「差参数」的分布
↓
⑥ 重复 ①~⑤，共 200~500 次
↓
⑦ 输出所有试验结果到 CSV，找出最优那组

🔬 五-B、Optuna 深入：TPE + MedianPruner 原理

TPE 是怎么「越猜越聪明」的？

核心思想：把所有已经试过的参数分成两组——效果好的前 25%（好参数组）和剩下的 75%（差参数组），然后分析两组各自的分布规律，下一次优先从「好参数组」分布中选值。
步骤做什么白话解释
1 按 MDA 排序所有已完成试验 把试过的参数按分数排个名次
2 前 25% 标为「好参数组」，后 75% 标为「差参数组」 成绩好的放一边，差的放另一边
3 分别计算两组各自的概率分布 看看「好参数」有什么共同特点，「差参数」有什么共同特点
4 新参数选在「好参数分布」概率高、「差参数分布」概率低的位置 尽量选在好学生扎堆、坏学生不来的区域
可视化理解：
试验 1~10：  ░░░░░░░░░░░░░░░░░░░░░░░░░░░ 到处试探
试验 11~30：      ░░░░░░░░░░░░░░░░░░░░░░ 范围缩小了
试验 31~50：          ░░░░░░░░░░░░░░ 聚焦到好区域
试验 51+：                ░░░░░ 接近最优

MedianPruner 怎么提前终止差的试验？

工作方式：LightGBM 每加 100 棵树时，Optuna 检查一次中间分数。如果当前试验的中间分数 差于所有已完成试验的中位数，直接终止这组试验。

🎛️ 六、9 个核心超参逐个详解

每一个参数都附带「调大了会怎样」「调小了会怎样」和「典型范围」。

参数名 一句话含义 详细解释 范围 调太大会？ 调太小会？
learning_rate 学习率 / 步长 每棵新树对最终预测的贡献权重。学习率小 = 每步走得少 = 需要更多树 = 更稳健但更慢 0.01~0.3 可能跳过了最优解，震荡不收敛 收敛太慢，需要很多棵树才能学好
num_leaves 叶子节点数 每棵树最多有多少个叶子节点。叶子越多 = 树越复杂 = 能捕捉更精细的模式 16~256 过拟合 记住噪声而不是规律 欠拟合 模型太简单，学不到复杂关系
max_depth 树最大深度 树从根到叶子最多经过几层。和 num_leaves 配合使用，一起控制模型复杂度 3~12 过拟合 欠拟合
min_child_samples 叶子最少样本数 一个叶子节点至少包含多少条训练数据。值越大，树越不容易在数据稀疏的地方分裂 5~100 欠拟合 阻止了有用的分裂 过拟合 可能在只有几条数据的地方也分裂
subsample 样本采样比例 每棵树只用多少比例的训练数据（随机抽取）。<1.0 能防止过拟合，增加泛化能力 0.5~1.0 1.0 失去正则化效果，可能过拟合 用太少数据，模型学不充分
colsample_bytree 特征采样比例 每棵树只用多少比例的特征列（随机抽取）。类似 subsample，但是按列采样 0.5~1.0 1.0 失去正则化效果 丢失重要特征，影响精度
reg_alpha L1 正则化 给损失函数加上「参数绝对值之和」的惩罚。会把不重要的特征权重压成 0，相当于自动做特征选择 1e-8~10 欠拟合 把有用特征也压成 0 了 正则化太弱，起不到防止过拟合的作用
reg_lambda L2 正则化 给损失函数加上「参数平方和」的惩罚。让所有参数都尽量小但不会变 0，更平滑地防止过拟合 1e-8~10 欠拟合 正则化太弱，可能过拟合
n_estimators 树的数量 一共建多少棵树。配合 learning_rate：学习率小就需要更多树 100~2000 训练慢、可能过拟合（但通常配合小学习率就还好） 学习不充分，欠拟合
关键理解：这些参数不是独立的——它们相互影响。比如 num_leaves 很大时就需要加大 reg_alpha/reg_lambda 来防止过拟合；learning_rate 很小就需要更多的 n_estimators。Optuna 的价值就在于能自动找到这些参数之间的最佳平衡点。

📏 七、评估指标：MDA / MAPE

我们需要用数字来衡量「预测得准不准」。两个指标从不同角度衡量预测效果。

指标全称公式含义大白话解释
MDA Mean Directional Accuracy
平均方向准确率		 预测方向正确的天数 ÷ 总天数 × 100% 只关心「方向预测对了吗」——预测明天涨、实际也涨，就计为正确。MDA 越高越好，范围 0%~100%。MDA=60% 表示 10 次里有 6 次涨跌方向预测对了。优点：直接反映模型对趋势方向的判断能力。缺点：不关心涨跌幅度的大小。
MAPE Mean Absolute Percentage Error
平均绝对百分比误差		 所有 |(预测值-真实值)/真实值| 的平均 × 100% 用百分比表示误差。MAPE 越低越好。MAPE=5% 意思是平均每次预测偏差 5%。优点：百分比直观，不同数据集之间可比较。缺点：真实值接近 0 时数值会爆炸。
MDA 的计算细节：方向正确 = sign(预测值t - 真实值t-1) == sign(真实值t - 真实值t-1)。只要预测的「变化方向」（涨/跌）和实际的「变化方向」一致就算对，涨跌幅度不重要。
MDA 和 MAPE 为什么搭配使用？MDA 衡量「方向对不对」，MAPE 衡量「数值准不准」。两者互补：一个模型可能方向全对但数值偏差很大（MDA 高、MAPE 差），也可能数值很准但方向经常错（MAPE 好、MDA 低）。同时看两个指标才能全面评估模型好坏。
Optuna 优化注意事项：Optuna 默认最小化目标函数，而 MDA 是越大越好。因此在定义 objective 函数时，如果以 MDA 为优化目标，需要返回 1 - MDA 或 -MDA 来转换为最小化问题。MAPE 本身就是越小越好，可以直接作为目标。

🎯 七-B、objective(trial) 函数详解

一句话：objective(trial) 是你写给 Optuna 的「评分函数」——Optuna 给你一组超参数，你训练模型后返回一个分数，Optuna 想办法让这个分数越小越好。

它做了三件事

① trial 建议一组超参
num_leaves = trial.suggest_int("num_leaves", 15, 255)   → 比如这次猜 128
learning_rate = trial.suggest_float("learning_rate", 0.01, 0.3, log=True)   → 比如猜 0.05
...（共 9 个参数）

② 用这组参数训练 LightGBM，在验证集上算 MDA
model = LGBMRegressor(num_leaves=128, learning_rate=0.05, ...)
model.fit(X_train, y_train)
y_pred = model.predict(X_val)
mda = 计算方向准确率(y_val, y_pred)   → 比如 MDA = 0.62

③ 返回分数（Optuna 会最小化这个分数）
return 1.0 - mda   → 返回 0.38

为什么返回 1 - MDA 而不是直接返回 MDA？

Optuna 的设计是「分数越小越好」（最小化），但 MDA 是「越大越好」。所以用 1 - MDA 做转换：
MDA返回给 Optuna 的值Optuna 的判断
0.621 - 0.62 = 0.380.38 < 0.45，所以选这个 ✓
0.551 - 0.55 = 0.45分数更高，不选 ✗

trial 对象是什么？

一句话：trial 是 Optuna 每次试验时专门为你创建的一个「参数顾问」对象——它的核心工作就是根据之前的历史经验，给你建议这一轮该试什么参数值。
注意一个关键点：
每一次 trial 都是独立的对象！
  • 第 1 次试验 → trial_1 对象
  • 第 2 次试验 → trial_2 对象
  • 第 200 次试验 → trial_200 对象

每个 trial 对象只负责当前这一轮的参数建议，但它能「看到」前面所有 trial 的历史结果。

trial.suggest_xxx 的工作原理（三步理解）

步骤做什么大白话
1 接收历史数据 trial 对象内部能看到前面 1~N 次试验用了什么参数、得到了什么 MDA 分数
2 TPE 算法计算 把历史分成「好参数组」和「差参数组」，计算两组的概率分布
3 建议新值 从「好参数分布」概率高、「差参数分布」概率低的区域抽一个值返回给你

trial 的核心方法详解

方法参数类型作用我们代码中的示例
suggest_int(name, low, high) 整数 在 [low, high] 范围内建议一个整数 trial.suggest_int("num_leaves", 15, 255)
→ 比如返回 128
suggest_float(name, low, high) 浮点数 在 [low, high] 范围内建议一个浮点数（均匀分布） trial.suggest_float("subsample", 0.5, 1.0)
→ 比如返回 0.82
suggest_float(name, low, high, log=True) 浮点数 在 [low, high] 范围内建议一个浮点数（对数分布） trial.suggest_float("learning_rate", 0.01, 0.3, log=True)
→ 比如返回 0.047
为什么 learning_rate 需要 log=True？
学习率 0.01 和 0.02 之间的差距，比 0.2 和 0.3 之间的差距重要得多。用对数尺度可以让小值区域被更密集地采样。

trial 的其他有用方法

方法作用示例
trial.set_user_attr(key, value) 给当前 trial 附加额外信息（不会影响优化） trial.set_user_attr("mda", 0.62)
trial.set_user_attr("mape", 0.15)
trial.number 当前是第几次试验（从 0 开始） 第 5 次试验 → trial.number == 4
trial.params 当前 trial 已经选好的所有参数字典 {'num_leaves': 128, 'learning_rate': 0.05, ...}
trial.suggest_xxx 不是纯随机抽——它背后是 TPE 算法在运作，会根据之前试验的结果越来越聪明地选值。第 1 次可能是乱猜，第 200 次就已经非常聚焦了。

直观理解：trial 的「学习曲线」

第 1~10 次试验：「盲人摸象」
  trial：我不知道哪个区域好 → 到处乱试
  num_leaves 可能是 20、200、50、180... 分布很散

第 11~50 次试验：「找到方向」
  trial：看起来 num_leaves 在 80~150 之间效果比较好 → 往这个区域聚焦
  num_leaves 更多出现在 80~150 之间

第 51~200 次试验：「精准搜索」
  trial：num_leaves 在 110~130 之间表现最好 → 在这个小范围内精细搜索
  num_leaves 可能是 112、118、125、121... 非常集中
重要 1：suggest_xxx 必须每次都按相同顺序调用！
如果你在 objective 函数里一会儿先 suggest num_leaves，一会儿先 suggest learning_rate，Optuna 会搞混的。必须保持顺序一致。
重要 2：trial 对象每次都传，但方法按需调用！
trial 对象身上有很多方法可用，但你不需要把它们都用一遍：
  • suggest_int/suggest_float → 必须调用（用来建议超参数）
  • set_user_attr → 可选调用（用来记录额外信息）

看我们的代码：我们只用了 suggest_int（4次）、suggest_float（5次）、set_user_attr（3次），其他 trial 方法根本没碰。

完整的 trial 生命周期示例

第 50 次试验的 trial 对象：

# trial 出生了，它能看到前 49 次的历史 def objective(trial): # 第 1 步：trial 根据前 49 次的经验建议参数 num_leaves = trial.suggest_int("num_leaves", 15, 255) # → trial 想：前 49 次里 110~130 效果最好，这次猜 122 learning_rate = trial.suggest_float("learning_rate", 0.01, 0.3, log=True) # → trial 想：0.03~0.07 之间不错，这次猜 0.048 ... # 第 2 步：用这组参数训练模型 model.fit(...) # 第 3 步：算 MDA = 0.65 mda = 计算... # 第 4 步：把额外信息记在 trial 身上 trial.set_user_attr("mda", mda) trial.set_user_attr("mape", mape) # 第 5 步：返回目标值，trial 完成使命退场 return 1.0 - mda

理解了各自的工作原理后，看它们是怎么配合的。

┌──────────────────────────────────────────────────────────────┐
│  Optuna（外层指挥：决定试什么参数）                              │
│                                                                                  │
│  ① TPE 建议一组超参                                                   │
│      ↓                                                                    │
│  ② 传给 LightGBM                                                     │
│      ↓                                                                    │
│  ┌─────────────────────────────────────────────┐      │
│  │ LightGBM（内层士兵：用这组参数训练模型） │      │
│  │                                                           │      │
│  │ ③ 建第1棵树 → 算残差                          │      │
│  │ ④ 建第2棵树 → 修正残差 → 算中间MDA         │      │
│  │ ⑤ MedianPruner 检查：太差就终止 ────→ 回到①  │      │
│  │ ⑥ 建第N棵树 → 训练完成                    │      │
│  │ ⑦ 在 val 集上算最终 MDA                   │      │
│  └─────────────────────────────────────────────┘      │
│      ↓                                                                    │
│  ⑧ 把 MDA 返回给 TPE，更新「好/差参数」分布                │
│      ↓                                                                    │
│  ⑨ 回到①，下一轮更聪明地选参数                             │
│                                                                                  │
│  重复 50~200 次，取 MDA 最高的那组参数                      │
└──────────────────────────────────────────────────────────────┘

三个角色的分工

角色是什么职责比喻
Optuna (TPE) 外层指挥官 决定每一轮试什么参数，学习历史结果 总厨：根据之前的试吃反馈，决定下一锅怎么调味
LightGBM 内层士兵 接收参数，训练模型，产出预测和分数 厨师：按配方做菜，端出来给人试吃打分
MedianPruner 监工 中途检查，太差就喊停，节省算力 试吃员：尝一口发现咸得没法吃，直接倒掉

train / val / test 在这个流程中的用途

Optuna 优化阶段：
  train.csv → 训练模型（学习数据中的规律）
  val.csv → 验证效果（Optuna 根据这个分数调整搜索方向）
  test.csv → ❌ 完全不碰！（最后的考试卷，调参时不能偷看）

复现推理阶段：
  train.csv + val.csv → 合并后，用最优参数重新训练
  test.csv → 最终考试，得出真正的 MDA/MAPE
为什么 test.csv 全程不能碰？如果你在调参过程中看了 test 的分数，然后根据这个分数回去调整参数，就等于「考前偷看了答案再复习」——最后考出来的高分是假的，因为你的参数是为这套特定考题优化的，换一套新题就不灵了。

🔄 八、为什么要复现？随机种子是什么？

为什么需要「可复现」？

  • 如果每次运行结果都不一样，你无法判断「改了参数」还是「运气好」导致效果变好
  • 团队协作时，别人跑你的代码应该得到完全相同的数字
  • 上线部署时，你需要确保训练环境和生产环境的模型一致
  • 学术/工作中，结果不可复现 = 结果无效

「随机」到底从哪来？

机器学习中有很多看起来「随机」但其实可以控制的地方：

随机来源影响什么怎么控制
Python 内置 random数据采样、打乱顺序random.seed(42)
NumPy 随机数数组操作中的随机np.random.seed(42)
LightGBM 训练特征采样、数据采样、树的构建random_state=42, deterministic=True
Optuna 采样器每次建议哪组参数TPESampler(seed=42)
关键技巧：所有地方用同一个种子数字（比如 42），并且把 LightGBM 的 deterministic=True 打开，就能确保每次运行结果完全一致。我们的复现标准是两次运行的预测值差异 < 1e-6（小于百万分之一）。
