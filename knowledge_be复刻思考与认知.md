# knowledge_be 复刻的思考与认知

> 搭完这套 RAG 检索之后的一些反思。核心想通了一件事：RAG 不是「一个更强的搜索框」，而是**给 AI 外挂一座可秒查的图书馆**——把「知识」从模型权重里搬出来，变成可更新、可溯源、可控的外部记忆。而且本指南只做 RAG 的「R」（检索）那半，「G」（生成）是 agent_service 的事。

---

## 一、想通的核心：RAG 是「外挂记忆」工程，不是「更强的搜索」

很多人（包括之前的我）以为 RAG 就是「给 AI 加个搜索功能」。搭完才发现这个理解太浅——RAG 的本质是**改变 AI 知识的来源**。

对比一下两种知识：

- **参数化记忆（直接问 LLM）**：知识在训练时被「烧」进模型权重里。静态、固化、不可更新（要更新得重新训练）、不可溯源（它说「铁矿石库存下降」你不知道它从哪看来的）、会幻觉（没见过也能编）。
- **检索记忆（RAG）**：知识放在外部库里，AI 回答前先去查。可更新（丢篇新文档进去就更新了）、可溯源（每条结果带来源文件）、可控（库里没有的它就不会硬编）。

> 一句话：**直接问 LLM 是让它「凭记忆答」，RAG 是让它「先查资料再答」。** 前者的知识锁在权重里，后者的知识放在图书馆里。

所以搭 RAG 的核心动作不是「调搜索算法」，而是「**给 AI 造一座外部记忆库**」——把领域知识从「模型内部、不可见、不可改」搬成「模型外部、可见、可改」。

但这里有个**边界要先划清**：RAG = Retrieval-Augmented Generation（检索增强**生成**）。这套 knowledge_service 只做了「检索（Retrieval）」那半——从库里捞出最相关的片段。「生成（Generation）」那半（把片段喂给 LLM 写答案）是 agent_service 的事。**先把检索那半搞透，生成只是把检索结果拼进 prompt。** 这份指南和那份思考，都聚焦在「R」。

---

## 二、「检索质量」靠四层设计保证

和 agent 的「四层约束」对应，这套 RAG 的检索质量也是**层层叠叠**的，但不是「约束」，是「**每层用一种机制保证一个质量维度**」：

| 质量层 | 用什么机制 | 保证什么 | 代价 / 换来什么 |
|---|---|---|---|
| **切片层** | 递归字符切片（chunk_size=500 + overlap=50）| 粒度——返回又准又短、能喂进 LLM 的片段 | 片段不能太碎（丢上下文）也不能太粗（混入无关）-> 换来精准定位 |
| **召回层** | 双路召回（KNN 向量 + BM25 关键词）+ RRF 融合 | 召回率——别漏掉相关片段 | 算两遍比算一遍贵 -> 换来语义+字面双覆盖 |
| **精排层** | cross-encoder（bge-reranker）对候选逐对重打分 | 准确率——top1 是真正最相关的 | 慢（每对跑一次模型）-> 换来深度理解 |
| **容错层** | 两层优雅降级（embed 挂->纯 BM25；rerank 挂->融合分）| 可用性——崩了也不空手而归 | 次优结果顶上 -> 换来服务永远尽量有输出 |

每一层都是用「一种代价」换「一个质量维度」。这就是工程——**工程不是追求单点最优，而是让多个维度同时及格**。只追求召回率会引入噪声，只追求精度会漏召回，只追求速度会牺牲准确，只追求准确一崩就全崩。四层叠起来，粒度、召回率、准确率、可用性才同时有保障。

> 把这四层和「一次检索」对照：你查一句话，先经**召回层**（双路捞几十个候选）-> 再经**精排层**（cross-encoder 重排挑出 topK）-> 中途任何一步崩了有**容错层**顶上。而这一切能成立的前提，是入库时**切片层**把文档切成了可比较的小块。

## 三、两段式 + 双路：承认「单点不够」，用分层和互补换「又要快又要准又要稳」

这是搭完之后最想通的设计哲学。新手直觉是「**用最强的一个模型，从头检索到底**」。真实工程恰恰相反——它**故意不用一个模型到底，而是把活拆成粗筛和精排两段，每段用不同强度的模型**：

### 1. 两段式检索：快的粗筛 + 慢的精排

- **召回用 bi-encoder（bge-m3）**：query 和 doc 各自独立编码成向量，doc 可预编码存库，查询只编 query，**秒级**从 237 个 chunk 里粗筛出几十个。代价是 query 和 doc 没互相「看到」，理解浅。
- **精排用 cross-encoder（bge-reranker）**：query 和 doc 拼成 pair 一起进模型，attention 让两者词词交互，**理解深**。代价是每对都要跑一次模型，慢。

为什么不能直接用 cross-encoder 检索全部 237 个 chunk？因为 237 次完整模型推理太慢。所以**先用快的粗筛出 top30，再用慢的精排挑 topK**。

> 这和 agent 复刻的「用约束换可控」是同构的——**都是用「分层/分工」换「既要又要」**。agent 用「流程约束」换「可控」；RAG 用「两段分层」换「又快又准」。单点最优在工程里几乎总是不可行的，因为现实约束（速度、成本、稳定性）是多元的。

### 2. 双路召回：承认每种检索都有盲区

召回层不是「选一个最好的检索方式」，而是**向量 + BM25 两路都跑再融合**：

- **向量（KNN）**：懂语义，「库存减少」能查到「库存下降」。盲区是漏精确专有名词（「LightGBM」可能被泛文档稀释）。
- **BM25（关键词）**：精确匹配，「LightGBM」三个字母必须出现才命中。盲区是不懂同义，查「库存下降」查不到只写「库存减少」的文档。

两路**互补**：向量补 BM25 的「不懂同义」，BM25 补向量的「漏专有名词」。然后用 **RRF** 融合——RRF 的聪明之处是**不看分数、只看排名**（`1/(k+rank+1)`），因为两路分数量纲不可比（一个是 0~1 余弦，一个是无界词频分）。两路都召回的「共识 chunk」双倍加分，自然冒泡到最前。

> 双路召回的本质是**承认单点有盲区**。这不是「哪个更强」的选择题，是「承认没有银弹、用互补覆盖盲区」的工程观。世界上没有一种检索方式能同时完美处理语义同义和精确术语，那就两种都用、取共识。

### 3. 优雅降级：真实工程的标志

容错层的设计最能看出「这是生产服务还是学生作业」：

- **embedding 失败 -> 退纯 BM25**：bge-m3 挂了，BM25 不依赖模型（只靠 jieba+词频），照样按关键词出结果。
- **rerank 失败 -> 用 RRF 融合分**：reranker 抽风，候选还在，按融合分排（粗一点），不空手。

学生项目崩了就崩了、返回 500；生产服务崩了要能用次优方案顶上。**模型推理总会偶发失败（OOM、加载慢、权重损坏），不能让一次抽风让整个 search 返回 500。** 每层都设 fallback，是工程标配。

> 两段式、双路、降级——这三件事合起来，反映的是同一个哲学：**不押注单点，用分层和互补对冲不确定性**。这和 agent 复刻里「不押注 AI 的自由、用约束对冲不可预测」一脉相承。

## 四、搭建知识库的意义：给 AI 外挂可溯源、可更新、可控的领域记忆

这是最大的收获。agent 复刻让我想通「问答可以工具化」；知识库复刻让我想通「**AI 的领域知识可以外挂**」。

### 1. 可更新：加文档就更新知识，不用重训模型

直接用 LLM，要让它懂「本周铁矿石库存」，得微调或重新训练——成本极高、周期长。RAG 不用：**往 `data/corpus` 丢一篇新报告，重新 ingest，知识就更新了**。知识的更新频率从「训练周期（周/月）」降到「入库周期（秒/分钟）」。

### 2. 可溯源：每条结果带来源，不再是「AI 说」

LLM 凭记忆答「铁矿石库存下降 5%」，你不知道它从哪看来的，可能是幻觉。RAG 检索返回的每条片段都带 `fileName/source/score`——**「这个结论来自 report_20260709.md 的第 3 个 chunk，相关度 0.87」**。从「AI 说」变成「AI 引用某文档说」，可信度天差地别。

### 3. 可控：白盒，能调能改能验证

托管知识库把整条链路当黑盒，出问题不会调、不知为什么准/不准。自搭的每个环节代码都在眼前：chunk_size 改大改小看效果、reRank 开关对比、降级亲手触发。**不仅会出结果，还懂每个结果怎么来的。**

### 4. 平民化：不用微调就能让 AI 懂一个领域

这是最关键的意义。让 LLM 懂「铁矿石预测」这个垂直领域，传统路径是微调——需要数据、需要算力、需要 ML 经验，门槛极高。RAG 把这个门槛砸了：**丢几篇领域文档进库，任何 LLM 立刻「懂」这个领域**。领域知识的注入，从「ML 工程师的活」变成「文档管理的活」。

> 我们搭知识库，本质是**把「让 AI 懂一个领域」的专业门槛，从「会微调」降到了「会整理文档」**。开发者吃下检索链路的复杂度，换业务侧一个「丢文档就能用」的知识库。

## 五、透明 vs 省事：自搭 vs 托管的权衡

不是所有场景都该自搭。这是个权衡，和 agent 复刻的「约束 vs 自由」一样：

**自搭换来**：透明（每环代码可见）、可控（能调能改）、能学（懂每个结果怎么来的）、能复刻（语义行为可对齐）。

**自搭的代价**：开发成本（要写切片/编码/检索/重排/降级）、运维成本（要部署 Chroma、管模型权重）、性能成本（本地 CPU rerank 是瓶颈）。

所以：

| 场景 | 该用什么 |
|---|---|
| 一周内上线客服问答 / 产品导向 / 不要求懂原理 / 文档量大要弹性 | **托管 KB**（火山方舟等，省事、黑盒） |
| 学 RAG 怎么工作 / 复刻已有服务 / 要调优要溯源 / 文档量小可控 | **自搭**（透明、可控、能学） |

一句话：**托管 KB 是「能用」的最短路径，自搭是「懂」的唯一路径。** 这次「项目复现」的初衷是学，所以必须自搭。但别把自搭当信仰——做产品时托管完全合理。

> 顺便澄清一个之前问过的点：火山方舟托管的 rerank **本质也是 cross-encoder**（和本地 bge-reranker 同类，都要 query+doc 一起进模型打分）。所以「用托管的」≠「学到了 cross-encoder」，你只是换了个黑盒 cross-encoder。本地用 bge-reranker，你能看见它每一步在干什么。

## 六、复刻的标准：语义对齐，不是技术栈对齐

原版 `knowledge_be` 是 Java + ES + MyBatis + RocketMQ + JanusGraph 的大服务（845 行主链路）。复刻版是 Python + Chroma + rank_bm25 的小服务。**凭啥叫「复刻」？**

想通的标准是：**复刻的不是技术栈，是 RAG 语义链路和接口契约**。

| 维度 | 原版（Java） | 复刻（Python） | 算复刻吗 |
|---|---|---|---|
| RAG 链路 | 召回->融合->重排->降级 | 同 | ✅ 对齐 |
| 三模式 | knn/bm25/mix | 同 | ✅ 对齐 |
| 降级逻辑 | embed 挂->纯 BM25 | 同 | ✅ 对齐 |
| 三段耗时日志 | embedding/search/reRank | 同 | ✅ 对齐 |
| 输出契约 | KnowledgeSearchVO 字段 | `_to_vo` 同字段 | ✅ 对齐 |
| **语义模型** | **bge-m3 + bge-reranker** | **同款 bge** | ✅ **对齐（根）** |
| 存储引擎 | ES | Chroma | ❌ 简化（可接受） |
| 持久层 | MyBatis-Plus | 无 | ❌ 简化 |
| 切片 | 段落/表格/长度多策略 | 递归字符单一策略 | ❌ 简化 |
| 基础设施 | RocketMQ/JanusGraph/S3 | 全跳 | ❌ 简化 |

**判断标准是「是否影响 RAG 语义行为」**：链路结构、输出契约、耗时日志、语义模型——这些影响「同样的 query 对同样的文档得到同样的相关度判断」，必须对齐。技术栈（ES/Chroma、Java/Python）、基础设施（MQ/图库）、运维细节——这些只换存储/调度方式、不改检索语义，可简化。

> **复刻成立的根，是「用 bge 同款模型」**。因为检索语义由 embedding/rerank 模型决定，原版用 bge、复刻也用 bge，同样的 query 对同样的文档会得到同样的相关度判断。换 doubao-embedding 等其他模型，语义就偏了，复刻名存实亡。**技术栈是表象，语义行为一致才是实质。** 验证：hit@5=88%、reRank=1 比 reRank=0 更准、降级生效——语义行为和原版对齐，复刻成立。

## 七、一次检索的完整旅程：从类比到实际链路

前面几节讲「为什么」，这一节补「实际怎么跑」——把一次检索从进来到出结果的完整路径串一遍，先类比，再对照真实代码。第二节那四层设计到底是哪句话、哪个函数在执行，看完这条链路就落地了。

### 1. 类比：你去图书馆借一本书

```
你说一句话「铁矿石库存预测准确率」
  │
  ▼
①馆员把你的话也贴个标签（query embedding）-> 变成能比较的向量
  │   (若贴标签的机器坏了 -> 直接走关键词检索，降级 1)
  ▼
②两路找书：
  │  向量路：按标签近找（KNN，懂语义）
  │  关键词路：按字面找（BM25，精确术语）
  ▼
③RRF 融合两份排名 -> 一份共识排名（两路都命中的冒泡到前）
  │
  ▼
④(可选)按日期/来源筛一筛（_filter 后处理）
  ▼
⑤精排员把 top20 候选每页都细看一遍（rerank cross-encoder）-> 重排
  │   (若精排员请假 -> 用融合分排，降级 2)
  ▼
⑥裁到 topK，贴上来源标签（_to_vo）-> 递给你
```

### 2. 实际链路（带文件位置）

```
HTTP POST /knowledge-base/report/search (routes.py)
  │  schemas.py 用 Pydantic 校验 content/topK/reRank/mode/日期
  ▼
search.search() (retrieval/search.py)
  │
  ▼
①embed(content) (pipeline/embedder.py)
  │  get_model() lru_cache 懒加载 BGEM3FlagModel -> encode -> normalize -> 1024 维
  │  try/except: 失败则 emb_ok=False（降级 1 触发点）
  ▼
②双路召回（pool_k = max(topK*3, 30) 多取留余量）:
  │  bm25.get_index().search(content) (retrieval/bm25.py)
  │    └─ jieba 分词 + BM25Okapi 词频打分（首次从 store 拉全量建索引）
  │  vector.search_vec(qvec) (retrieval/vector.py)
  │    └─ store.query_knn (pipeline/store.py) Chroma cosine，1-dist 转相似度
  │  if not emb_ok 或 mode=="bm25": merged = bm_hits          ← 降级 1
  │  elif mode=="knn": merged = vec_hits
  │  else: merged = hybrid.rrf([vec_hits, bm_hits]) (retrieval/hybrid.py)
  │           └─ 1/(k+rank+1)，k=60，共识 chunk 加分
  ▼
③_filter (search.py): 按 publishTime/sourceType 后处理精筛
  ▼
④candidates = merged[:max(topK*2,20)]
  │  if reRank==1: rerank_fn(content, docs) (pipeline/reranker.py)
  │    └─ AutoModelForSequenceClassification（绕开 FlagReranker，Q6 的 bug 修复）
  │       pairs=[[query,doc]...] -> sigmoid -> 重打分 -> sort
  │    try/except: 失败则保留融合分不重排（降级 2 触发点）   ← 降级 2
  ▼
⑤candidates[:topK] -> _to_vo (search.py): 对齐 KnowledgeSearchVO 字段
  ▼
返回 (hits_vo, timings) -> JSON 回前端/agent
  │  timings = {embeddingTimeCost, searchTime, reRankTimeCost, totalHits}
```

### 3. 这条链路里，第二节的四层设计正在动

- **切片层**：这条链路能成立，前提是入库时 `splitter.py` 已把文档切成 237 个可比较的 chunk（Q1）。没有切片，检索单位就是整篇文档，精度和长度都崩。
- **召回层**：②里 `bm25 + vector + rrf` 三件套在跑，双路互补 + 共识冒泡（Q3）。
- **精排层**：④里 `rerank_fn` 用 cross-encoder 对候选逐对重打分（Q4）。
- **容错层**：①的 `try/except`（降级 1）和④的 `try/except`（降级 2）在兜底（Q5）。

> 把这条链路和第二节对照着看就明白：所谓「四层设计」不是抽象口号，而是这条路径上 embed、双路、RRF、filter、rerank、_to_vo 每一处函数，都在替检索质量保证一个维度。

### 4. 两个容易混的细节（读源码时纠偏）

- **store 的距离要 `1 - dist` 翻转**：Chroma 返回 cosine **距离**（0=最像），但排序要**相似度**（越大越好），所以 `score = 1 - distance`。读源码不留意这个，分数方向会搞反。
- **BM25 依赖 store、不是独立存储**：`bm25.get_index()` 首次调用时从 Chroma 拉全量 chunk 建索引。**必须先 ingest 入库，BM25 才有数据**——这就是 `ingest.py` 末尾要 `bm25.reset_index()` 的原因。BM25 不是独立数据库，是「从向量库派生的内存索引」。

## 八、回到项目复现2：这套知识库给了它什么

项目复现2 加了 agent 之后，报告的数据来源是 `fetch_data`——读 `07_model_training/outputs` 里的模型预测和评估指标。这些是**模型算出来的数字**，对，但单薄：只有「预测值是多少、MDA 多少」，没有「这个数字在行业里意味着什么、历史上什么情况」。

加这套知识库之后，agent 多了一个 `fetch_knowledge` 工具（`mcp_server` 桥接 :8092）。报告从「只有模型预测数据」扩展到「**预测数据 + 领域分析文档**」：

- 预测说「库存下降」，知识库检索出「历史上库存下降通常伴随价格上涨」的分析片段——报告从报数字变成**有背景、有对照、有解读**。
- 每条结论能溯源到具体文档——报告从「AI 说」变成「**引用某报告说**」。
- 丢篇新行业报告进库，下次报告立刻能用——知识**可更新**，不用重训模型。

> 这套知识库给项目复现2 加的不是「检索能力」（那个 ES 也能干），而是**给 agent 的报告外挂了一座可溯源、可更新、可控的领域记忆**。让报告从「只有冷冰冰的预测数字」，变成「预测 + 事实依据 + 可溯源解读」。两份复刻在此交汇：**知识库复刻提供「检索能力」，agent 复刻提供「调用检索的研究流程」**——`fetch_knowledge` 工具把两者缝起来。

## 九、最后的一句话

搭知识库这件事，表面上是在写切片、写编码、写 BM25、调 rerank，骨子里是在做一道选择题：

**你要 AI 靠「参数记忆」答，还是靠「检索记忆」答？**

- 选参数记忆 -> 直接问 LLM，享受万能但不可溯源、不可更新、会幻觉
- 选检索记忆 -> 搭 RAG，用一座外部图书馆换可溯源、可更新、可控

还有一道连带的选择题：**你要「透明」还是「省事」？**

- 选省事 -> 托管 KB，把 RAG 当黑盒调 API
- 选透明 -> 自搭，每个零件都看得见摸得着

当你开始选「让 AI 先查资料再答」、并且选「自己搭来看清每个零件」的那一刻，你就从 AI 的**使用者**，变成了 AI 记忆的**建造者**。这次复刻，就是跨过那道门——而且和 agent 复刻那道门是同一扇：agent 让你从「用 AI」跨到「造 AI 产品」，知识库让你从「用 AI 的记忆」跨到「给 AI 造记忆」。

## 附：学习指南的编排逻辑（为什么这么排）

> 这份思考文档配套的 `knowledge_be复刻学习指南.html`，编排逻辑是对标 `agent复刻学习指南.html` 的清晰度做的。这里把「为什么这么排」总结下来，既是编排备忘，也是「怎么把一个技术系统讲清楚」的方法论沉淀。

### 1. 对标的三条清晰度原则

agent 指南读起来清晰，拆解后发现靠三件事，knowledge_be 指南全量对齐：

- **①一条主线追到底**：agent 指南用「一次请求的时间线」（coordinator->planner->researcher->reporter）串起所有问题。knowledge_be 指南对应「一次检索的时间线」（embed->双路->RRF->rerank->降级），问题严格按执行顺序排，不跳跃。
- **②概念随用随讲**：agent 指南从不开一个独立的「什么是 MCP」问题，而是把 MCP 折进 researcher 装工具箱那一步。knowledge_be 指南同样把「bge-m3 是什么」折进 Q2「块怎么变可查」的编码步骤，**不单独开概念问题打断流水线叙事**。
- **③一个比喻贯穿**：agent 指南用「研究咨询公司」比喻贯穿全文。knowledge_be 指南用「**图书馆**」比喻贯穿——入库=编目上架、检索=借阅查询，且开篇就点明「和 agent 复刻不同，这里有两条流水线」。

### 2. 关键差异：从「一条流」到「两条流水线」

agent 复刻只有「一条请求流」（请求进来->出报告），所以 8 个问题天然是一条时间线。但 **RAG 有两条流水线**：先建馆（入库，离线跑一次），再查库（检索，每次实时跑）。这是和 agent 指南最大的结构差异。

所以指南用**三个 Part 分隔**显式切开：

| Part | 主题 | 问题 | 颜色 |
|---|---|---|---|
| **PART 1** | 入库·把图书馆建起来（离线） | Q1 切块 / Q2 编码存储 | 蓝绿 |
| **PART 2** | 检索·你来查一句话（在线，按执行顺序） | Q3 双路召回 / Q4 重排 / Q5 降级 | 蓝紫 |
| **PART 3** | 工程反思（调试·选型·复刻） | Q6 bug / Q7 选型 / Q8 复刻 | 紫粉 |

没有这个分组，8 个问题平铺，读者看不出「建馆」和「查库」的切换，就会觉得乱。**Part 分隔让「两条流水线」在视觉上一目了然。**

### 3. 8 个问题的统一模式

每个问题都是同一套结构，降低认知负担：

```
驱动问题（一句话点出矛盾/困惑）
  → 一句话答案（先给结论）
  → 代码（锚定到真实文件）
  → 动手实验（亲眼验证）
  → 自测题（折叠答案，检验理解）
```

这套模式从 agent 指南继承，保证「每个概念都锚定到马上能动手验证的东西」，而不是飘着讲。

### 4. 编排上踩过的坑（初版不清的原因）

初版 knowledge_be 指南不如 agent 指南清晰，诊断下来三个原因，重写时全改了：

- **坑 1：概念独立成 Q，打断叙事**。初版把「bge-m3 是什么」单开一个问题，读者正跟着「块怎么变可查」走，突然被一个纯概念问题打断。重写：折进 Q2 编码步骤里随用随讲。
- **坑 2：没有两流分组**。初版 8 问平铺，看不出建馆/查库的切换。重写：加三个 Part 分隔。
- **坑 3：Q1 太满**。初版 Q1 把 load/split/embed/store 全塞一个问题。重写：拆成 Q1（只讲切块）+ Q2（讲编码存储），两个独立概念各得其所。

### 5. 第二阶段的递进逻辑

第一阶段（8 问）建立「地图」，第二阶段下到「发动机舱」，分 4 层递进：

- **L1 共享引擎**：每个模块都在调、但指南没打开的 3 个机制（config/logger/lru_cache 懒加载）。**先学语法再读句子**——不懂这些，管道代码读起来像外语。
- **L2 管道内部**：按数据流方向通读两条线（入库线 loader->splitter->embedder->store->ingest；检索线 embedder->bm25+vector->hybrid->reranker->search），每个模块盯「输入/调什么/产出」三列。
- **L3 被略过的子系统**：指南为讲清主干略掉的部分（_filter 过滤 / _to_vo 对齐 / FastAPI 层 / MCP 桥接 fetch_knowledge / loader HTML 抽文本），拓宽视野。
- **L4 动手改造**：改参数 + 跑 eval.py 闭环验证（先预言再验证，预言对了=懂了）。

L1 在前是因为：管道代码里全是 `lru_cache`、`os.getenv`、`log`、`store.count()`，不懂共享机制读管道就像读外语。先花半小时过 L1，L2 的管道代码就会「自己读自己」。

> 一句话总结编排心法：**先跑起来再说原理、一条主线追到底、概念随用随讲、一个比喻贯穿、动手验证每个概念、Part 分隔显式分组两条流水线。** 这套心法不绑定 RAG，换任何技术系统的学习指南都能复用。
