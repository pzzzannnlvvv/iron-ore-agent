## 阶段 1 · 电梯陈述与项目定位(开场 30 秒)

### 面试官会问
- 用一句话讲讲你这个项目?
- 你在里面具体做了什么?
- 为什么会想到做这个?

### 要弄懂的代码点
- 项目根 `README`、`agent复刻进度.md`(项目全貌与你的贡献边界)
- 三服务入口:`agent_service/main.py`(app / CORS)、`mcp_server/server.py`、`knowledge_service`

### 答法骨架
- **定位**:这是一个「铁矿石港口库存预测 + 自然语言问答」的垂直领域 AI Agent。底层是库存预测模型(给走势),上层是 deepresearch 风格的 agent——业务人员用自然语言问「最近库存趋势如何」「供需怎么看」,agent 自动拆解任务、调度工具(查数据 / 查知识库 / 跑预测 / 画图)、流式生成报告。
- **技术栈一句话**:LangGraph 编排多 agent + FastMCP 工具层 + RAG 知识库(bge + Chroma)+ FastAPI / SSE 流式输出,三个独立服务协作。
- **贡献**:复刻并跑通整条链路——从预测模型、agent 编排、RAG 接入到前端可视化界面,端到端联调验证(hit@5 ≈ 88%,agent 能正确引用知识库出报告)。

### 三层追问防线
- **Q:为什么不直接写 SQL 让业务查?**
  A:业务人员不会写 SQL;且「供需怎么看」这种问题要综合多个数据源 + 知识库 + 预测,agent 能自动拆解、按需调度、综合研判,比固定查询灵活得多。
- **Q:agent 和直接调一次 LLM 有啥区别?**
  A:单次 LLM 调用没有工具、没有多步规划;这个 agent 能拆任务、按需调工具、流式出报告,是 agentic(有自主规划与工具使用)而非单轮 QA。
- **Q:和市面上的 deepseek-research 啥区别?**
  A:通用研究 agent 没有领域工具;这个接了铁矿石专属工具(库存预测模型、行业知识库),是垂直领域 agent,回答更准、更可控。

## 阶段 2 · 整体架构(必问)

### 面试官会问
- 画一下你的架构?
- 三个服务为什么分开?能不能合成一个?
- 服务之间怎么通信?

### 要弄懂的代码点
- `agent_service/main.py`(CORS `allow_origins=["*"]`、路由注册)
- `agent_service/src/server/deepresearch.py:33` `chat_stream`(SSE 入口)
- `mcp_server/server.py`(8 个 `@mcp.tool`,FastMCP)
- `knowledge_service`(FastAPI,:8092)
- `agent_service/.env.dev` 的 `MCP_*_SETTINGS`(工具注册配置)

### 答法骨架
- **三服务**:`agent_service`(:5000,LangGraph 编排 + GLM 大模型)→ `mcp_server`(:17000,FastMCP 工具层,8 个工具)→ `knowledge_service`(:8092,RAG 向量库)。
- **通信链路**:前端 ↔ agent_service 走 **SSE 流式**;agent_service ↔ mcp_server 走 **MCP 协议**;mcp_server ↔ knowledge_service 走 **HTTP**(`fetch_knowledge` 内部 `httpx` 转发到 :8092)。
- **为什么分三层**:① 职责单一(编排 / 工具实现 / 知识库各自演进);② 工具层独立,加工具不动 agent;③ 知识库 Chroma 可独立重建 / 换 Milvus,不和 agent 同进程。

### 三层追问防线
- **Q:合成一个服务行不行?**
  A:技术上能,但工具实现和 agent 编排耦合,改工具要重启 agent;Chroma 重,不该和 agent 同进程。分开是为解耦和独立扩展。
- **Q:瓶颈在哪?**
  A:LLM 调用是主要延迟(秒级),工具调用和数据检索是次要;架构层不是瓶颈,模型推理才是。
- **Q:单点故障?**
  A:三个服务任一挂链路就断。生产化要加健康检查、重试、降级(知识库挂了,agent 能否只靠数据回答)。

## 阶段 3 · Agent 核心机制(主菜,重点)

### 面试官会问
- 为什么用 LangGraph,不用普通 while 循环调 LLM?
- 一个请求从进来到报告出来,经过哪些节点?
- coordinator 怎么决定交给 planner 的?
- planner 怎么拆任务?
- 听说 research_team 节点是空的?那调度在哪?

### 要弄懂的代码点
- `nodes.py:648` `coordinator_node`(`bind_tools([handoff_to_planner])`)
- `nodes.py:318` `planner_node`
- `planner_model.py:7` `StepType` / `:16` `Step` / `:29` `Plan`
- `nodes.py:1169` `research_team_node`(空,`pass`)
- `builder.py:22` `continue_to_running_research_team` / `:104` `add_conditional_edges`
- `nodes.py:1176` `_execute_agent_step` / `:1415` `_setup_and_execute_agent_step`
- `nodes.py:1476` `researcher_node`
- `nodes.py:1108` `reporter_node`

### 答法骨架
- **为什么 LangGraph 不用 while**:while 是「调一次 LLM → 看结果 → 硬编码决定下一步」的命令式控制流,状态散乱、难持久化、难恢复;LangGraph 把每步声明为图节点,状态在 `State` 里流转,天然支持 **checkpoint(断点续跑)、条件路由、可视化**。这个项目用 LangGraph 是为了把「协调→规划→研究→报告」这条复杂流程显式建模成图,而不是塞进一个 while 黑盒。
- **节点旅程**:coordinator(接单、决定要不要研究)→ planner(拆成 Step 列表)→ research_team(调度室,空壳)→ researcher(逐个 Step 执行,调工具)→ reporter(汇总出报告)。条件边控制流转。
- **tool-as-route(节点跳转的核心模式)**:coordinator 不是用 if/else 决定跳转,而是给 LLM 绑一个工具 `handoff_to_planner`,LLM 自己判断何时调用,调用即触发 `Command(goto="planner")`。**跳转由 LLM 决策,不是硬编码**。后面所有节点切换都是这个套路。
- **Plan / Step 结构**:planner 产出 `Plan{title, thought, steps:[Step]}`,每个 `Step{title, description, step_type, need_search, execution_res}`。`step_type`(RESEARCH/ANALYSIS/PROCESSING)决定路由,`need_search` 决定要不要调工具,`execution_res` 初始 `None`、执行后填结果。
- **空节点调度(全项目最反直觉的点)**:`research_team_node` 函数体就一句 `pass`。真正调度靠图的**条件边**:`continue_to_running_research_team` 检查 Plan 里有没有 `execution_res is None` 的 Step,有就路由到 researcher 执行下一个,researcher 执行完回到 research_team,再检查……**反馈式轮询**,直到所有 Step 执行完才进 reporter。**节点是空的,图结构承担了调度**——这是声明式控制流。

### 三层追问防线
- **Q:某个 Step 失败怎么办?**
  A:目前 `execution_res` 会记录失败信息,流程继续(容错);生产化要加重试 + 失败 Step 标记 + reporter 感知失败 Step 做说明。
- **Q:能并发执行多个 Step 吗?**
  A:当前是串行轮询(一次一个 Step)。LangGraph 支持 fan-out 并发,但 Step 间可能有依赖(后步用前步结果),需要 Step 里声明依赖——`planner_model.py:12` 的 `StepAssociation` 就是干这个的。
- **Q:为什么用空节点不直接 while?**
  A:空节点 + 条件边把「调度逻辑」从代码挪到图结构,可被 checkpoint、可视化、静态分析;while 是运行时黑盒。这是声明式 vs 命令式的取舍,也是用 LangGraph 的核心理由。

## 阶段 4 · 工具与 RAG(重点)

### 面试官会问
- 工具是怎么挂到 agent 上的?
- 不同视角(供应/需求/价格)工具箱不一样,怎么实现的?
- fetch_knowledge 的 RAG 是怎么做的?
- 为什么用 RAG 不直接把知识塞进 prompt?
- 召回效果怎么样?

### 要弄懂的代码点
- `mcps.py:7` `merge_mcp_tools`(按 `agent_name == node` 过滤)
- `mcp_server/server.py:46` `fetch_data` / `:122` `fetch_knowledge` / `:178` `model_predict` / `:188` `draw_chart`
- `knowledge_service`:bge embedding + Chroma 向量库 + 检索 API(:8092)
- `server.py:122` `fetch_knowledge` 内部 `httpx` 转发到 :8092

### 答法骨架
- **工具挂载(最小权限)**:`merge_mcp_tools` 把 mcp_server 的 8 个工具合并,但**按 `agent_name == node` 过滤**——coordinator、researcher、reporter 拿到的工具集不同。researcher 才有 `fetch_data`/`fetch_knowledge`/`model_predict`/`draw_chart`,coordinator 只有 handoff 类工具。每个节点只看到自己该用的工具,避免误调用。
- **视角切换**:`AgentType` 枚举(DEFAULT/SUPPLY/DEMAND/SD_BALANCE/PRICE/REPORT),不同视角对应不同工具配置(通过 `.env` 的 `MCP_*_SETTINGS`)。**注意工程债**:价格/供应/需求视角配置里引用了几个未实现的工具(`fetch_opinion_index` 等),所以前端这几个视角暂 disabled,只有通用 + 研报可用——诚实承认这部分是 demo 级。
- **RAG 全链路**:`fetch_knowledge(query)` → `httpx` 转发到 knowledge_service:8092 → bge 把 query 向量化 → Chroma 检索 topK →(可选 reRank)→ 返回拼接的知识片段给 researcher 作上下文。
- **为什么 RAG 不塞 prompt**:知识库量大(行业研报、数据说明),全塞 prompt 超上下文、成本高、噪声多;RAG 按需检索,只把相关的喂给 LLM,准且省。
- **召回效果**:联调实测 **hit@5 ≈ 88%**(top5 命中率),agent 端到端能正确引用知识库内容出报告。

### 三层追问防线
- **Q:召回率怎么测的?**
  A:构造一批 query + 标准答案文档,hit@5 = top5 里含正确文档的比例。88% 还有提升空间(reRank 调优、chunk 策略)。
- **Q:知识库怎么构建?**
  A:行业研报 / 数据说明文档 → 切 chunk → bge embed → 存 Chroma。可由 `ingest.py` 重建,所以 Chroma 数据不进 git。
- **Q:召回不准怎么排查?**
  A:看检索的 chunk 是不是相关 → 不行调 chunk 大小 / overlap → 再不行加 reRank → query 改写(用 LLM 扩展 query)。bge 选它是因为中文检索好、本地部署、免费。

## 阶段 5 · 流式输出与前端(工程亮点)

### 面试官会问
- 流式输出怎么实现的?为什么用 SSE 不用 WebSocket?
- 你说有 8 种事件,分别什么时候发?
- 前端怎么把一段段 chunk 拼成完整报告?

### 要弄懂的代码点
- `utils.py:18` `SSE_DONE` / `:310` `_make_event` / `:389` `_stream_graph_events` / `:453` `_process_message_chunk`
- `deepresearch.py:33` `chat_stream`(`StreamingResponse`,`text/event-stream`)
- `frontend/index.html`(`streamReport`、`handleEvent`、实时进度面板)

### 答法骨架
- **SSE 实现**:`chat_stream` 返回 FastAPI 的 `StreamingResponse`,`media_type="text/event-stream"`。`_stream_graph_events` 把 LangGraph 事件流转成 SSE 事件逐个 yield,结尾 yield `SSE_DONE`(`event: finish\ndata: [DONE]`)。
- **为什么 SSE 不 WebSocket**:SSE 单向(服务器→客户端)、走 HTTP、浏览器自动重连,足够 LLM 流式场景;WebSocket 双向、更重、要维护连接状态。LLM 输出是「服务端推、客户端只收」,SSE 是最简方案。
- **8 种事件**(`_process_message_chunk` 分流):
  - `message_chunk`:LLM 文本片段(reporter 写报告时逐 token 推)
  - `tool_calls`:agent 决定调工具(整体)
  - `tool_call_chunks`:工具调用的流式片段
  - `tool_calls_injection`:注入的工具调用信息
  - `tool_call_result`:工具执行结果返回
  - `interrupt`:中断(人工介入,本项目预留未用)
  - `error`:出错
  - `finish`:结束(配 `[DONE]`)
- **前端拼接**:`handleEvent` 监听 SSE 流按 event 类型分流——`message_chunk` 累加到报告区;`tool_calls`/`tool_call_result` 更新进度面板时间线;每个事件带 `langgraph_node`/`agent` 字段,前端据此显示当前在哪个节点。进度面板最终折叠成「用时 X · 调用 N 个工具」。

### 三层追问防线
- **Q:断线重连?**
  A:SSE 浏览器原生支持重连(带 Last-Event-ID);但 LangGraph 流是一次性的,生产化要配 checkpoint(`thread_id`),重连后从断点续。
- **Q:大报告内存?**
  A:前端逐 chunk 累加是字符串拼接,大报告别反复重渲染整个 DOM,要增量 append。
- **Q:为什么 SSE 特别适合 LLM?**
  A:LLM 生成是 token-by-token 的流,天然适合「服务端持续推」的 SSE;客户端不需要在生成途中发指令,单向足够。

## 阶段 6 · 数据与预测底子(证明不全靠 LLM)

### 面试官会问
- 你的预测模型是什么?效果怎么样?
- 特征工程怎么做的?
- 预测结果怎么进到 agent 里的?

### 要弄懂的代码点
- `05_feature_engineering/`(特征流水线,outputs 被 gitignore,可重新生成)
- `06_data_split/`(train/test 划分)
- `mcp_server/server.py:46` `fetch_data`(`pd.read_csv` 读预测结果)
- 最新训练进度见 `agent复刻进度.md`(以下指标以该文件最新值为准)

### 答法骨架
- **预测模型**:铁矿石港口库存预测,用历史库存 + 特征预测未来走势。核心指标 **MDA(方向准确率)**。
- **特征工程**:05 脚本做因子滞后、趋势波动、特征合并(03_factor_lags / 04_trend_volatility / 08_features_merged / 09_features_final),06 划分 train/test。大 CSV 被 gitignore(可由脚本重新生成)。
- **训练目标演进**:从 `multi_target` 训练目标向 `diff`(差分)目标迁移——`multi_target` MDA ≈ 52.36%(迁移进行中,以最新进度为准)。改 diff 是因为差分目标预测「变化量」,更稳定、更贴合业务关心的「涨多少/跌多少」,避开非平稳序列直接预测的偏差。
- **进 agent**:`fetch_data` 工具直接 `pd.read_csv` 读预测结果 CSV,把数据作为文本返回给 researcher,agent 据此解读。

### 三层追问防线
- **Q:MDA 指标怎么定的?**
  A:MDA = 预测方向(涨/跌)与实际方向一致的比例。库存预测更看重方向(决策用),不看重绝对值误差,所以用 MDA 不用 RMSE。
- **Q:为什么改 diff 目标?**
  A:原 `multi_target` 直接预测值,差分目标预测「变化量」,更稳定、更贴合业务关心的「涨多少/跌多少」,且能避免非平稳序列直接预测的偏差。
- **Q:数据质量?**
  A:港口库存数据有缺失/口径变化,特征工程里做了滞后对齐和异常处理;这是预测误差的主要来源之一。

> ⚠️ 这一阶段的具体指标和迁移进度,请以 `agent复刻进度.md` 最新值为准再核对——这是你的数据科学工作,你比我清楚最新训练结果。

## 阶段 7 · 取舍、坑、改进(高分项,区分候选人)

### 面试官会问
- 这个项目最大的技术挑战是什么?
- 踩过什么印象深的坑?
- 如果重做会怎么改?
- 项目有什么局限?

### 答法骨架(用真实经历,最加分)
- **网络坑**:推 GitHub 时直连间歇阻断(443 连不上 / 连接中断),本机无代理,最终靠 HTTP/1.1 + 大 buffer + 裸连重试解决。体现「在没有理想环境下解决实际问题」。
- **RAG 调优**:从初版召回不准到 hit@5 ≈ 88%,调了 chunk 策略 / reRank。体现迭代优化能力。
- **diff target 迁移**:预测目标从 `multi_target` 改 `diff`,是有意识的建模取舍,不是拍脑袋。
- **未实现工具的工程债**:几个视角引用未实现工具,前端 disabled 处理——体现对不完美状态的诚实认知和降级处理。

### 三层追问防线
- **Q:数据量翻 10 倍?**
  A:Chroma 换 Milvus/Qdrant;预测模型重训;agent 加缓存(同 query 不重复调 LLM)。
- **Q:多用户并发?**
  A:三个服务无状态可水平扩;LLM 调用是瓶颈,要限流 + 队列;LangGraph checkpoint 支持多 `thread` 并行。
- **Q:怎么产品化?**
  A:加鉴权、用量计费、结果缓存、知识库定期更新、监控告警、降级策略(知识库挂了走纯数据)。
- **Q:局限?**
  A:单轮报告(无多轮对话记忆)、几个视角未实现、预测模型精度有限、无并发/鉴权(目前 demo 级)。诚实说局限比硬吹更得分。
